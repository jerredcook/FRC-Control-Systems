Hardware lab

First contact with a
real robot.

You have the Python basics. Now prove that code you wrote can reach real hardware: blink an LED with a button press, then turn a wheel by hand and watch the encoder numbers stream back to your screen.

The mission

Two proofs, one little robot

This lab is about closing the gap between "my code runs" and "my code moves electrons in the real world."

The robot is the XRP, the same low-cost robot the course capstone drives in Part 9. It costs roughly $35 to $70, so check the current price. Your program does not get copied onto it. It runs in WPILib simulation on your laptop with RobotPy, and a WiFi link carries commands out to the XRP and sensor readings back, many times per second.

You are going to collect two proofs:

  • Output works: press the button on the XRP and your code blinks the onboard LED.
  • Input works: turn a wheel by hand and your code prints the encoder counts to the console.

Once both directions of that link work, everything else in this course is a variation: bigger outputs (motors), richer inputs (gyros, cameras), smarter code in between.

What you need

Gather three things

  • An XRP kit, assembled (the robot with its motors and sensors).
  • Charged batteries in the XRP's own pack. Weak batteries are the number one cause of mystery problems, so start fresh.
  • A laptop with RobotPy and WPILib installed, which you set up back in Part 1.
Heads up

Your laptop will join the XRP's own WiFi network for this lab, which means no internet while you are connected to the robot. Run your installs and open this page and the official guide before you switch networks, or use a phone for lookups.

Step 1

Create the XRP project

Make a new folder for the project, something like first-contact, and set up a RobotPy project in it the same way you did in Part 1, with a robot.py and a pyproject.toml. The XRP classes we use below, XRPMotor, XRPGyro, XRPOnBoardIO, and XRPReflectanceSensor, ship inside wpilib.xrp, so a standard RobotPy install already has them.

When it is time to run, the --xrp flag on robotpy sim tells the simulator to connect to the robot at its address, 192.168.42.1, instead of running purely on the laptop.

Follow the official guide for the exact commands

We teach the idea here. Versions and commands change, so follow the official WPILib XRP guide for the exact, current setup, including the precise RobotPy commands. It is kept up to date by the people who make the tools.

WPILib XRP guide ↗
What you should see

A project folder with robot.py and pyproject.toml, and robotpy sync finishing without errors. No robot needed yet, this step is all on the laptop.

Step 2

Power on and join the XRP's WiFi

  1. Put the charged batteries in and flip the XRP's power switch on.
  2. Give it about 30 seconds to boot.
  3. On your laptop, open the WiFi list and join the network named XRP-XXXX (the letters and digits differ per robot). The default password is xrp-wpilib.

This works like pairing bluetooth headphones: the robot broadcasts its own network and your laptop connects to it directly. There is no router in the middle.

What you should see

A power light on the XRP, the XRP-XXXX network in your WiFi list, and your laptop connected to it. Your laptop will probably warn "no internet." That is normal and correct, the XRP is a robot, not a router.

Step 3

Paste the first-contact code

Replace everything in robot.py with the file below. It is complete, paste-ready, and uses the same XRP hardware map the capstone uses: motors on channels 0 and 1, encoders on DIO 4 and 5 and 6 and 7. Note that RobotPy keeps the WPILib method names, like getUserButtonPressed and getAngle.

robot.py python
import wpilib
from wpilib import Encoder, Timer
from wpilib.xrp import XRPGyro, XRPMotor, XRPOnBoardIO, XRPReflectanceSensor


class MyRobot(wpilib.TimedRobot):
    def robotInit(self):
        # The USER button and LED on the XRP board, our first input and output
        self.io = XRPOnBoardIO()

        # Declared now to prove they construct; you drive them in the capstone
        self.left_motor = XRPMotor(0)
        self.right_motor = XRPMotor(1)

        # The wheel encoders, on the same DIO channels the capstone uses
        self.left_enc = Encoder(4, 5)
        self.right_enc = Encoder(6, 7)

        self.gyro = XRPGyro()
        self.reflectance = XRPReflectanceSensor()

        self.last_print = 0.0

    def teleopPeriodic(self):
        # Proof 1: hold the USER button and the LED blinks about twice a second
        pressed = self.io.getUserButtonPressed()
        blink_on = int(Timer.getFPGATimestamp() * 4) % 2 == 0
        self.io.setLed(pressed and blink_on)

        # Proof 2: print the sensors twice a second; turn a wheel and watch
        now = Timer.getFPGATimestamp()
        if now - self.last_print > 0.5:
            self.last_print = now
            print(
                f"L={self.left_enc.get()}  R={self.right_enc.get()}"
                f"  gyro={self.gyro.getAngle():.0f}"
                f"  line={self.reflectance.getLeftReflectanceValue():.2f}"
                f"/{self.reflectance.getRightReflectanceValue():.2f}"
            )


if __name__ == "__main__":
    wpilib.run(MyRobot)

Read it before you run it. teleopPeriodic runs about 50 times per second while the robot is enabled in teleop. The blink line is Part 2 math: Timer.getFPGATimestamp() is the seconds since the program started, multiplying by 4 and taking % 2 flips a boolean every quarter second. The LED only lights when the button is held and the blink boolean is true.

What you should see

No output yet, that is Step 4. If you want a dry check first, run robotpy sim without the robot: the Simulation GUI should open, which proves the file has no typos.

Step 4

Run it and blink the LED

  1. Confirm your laptop is still on the XRP-XXXX network.
  2. In the project folder, run robotpy sim --xrp. The Simulation GUI opens, and your program connects to the XRP over WiFi.
  3. In the GUI, set the robot state to Teleop, then click Enable.
  4. Press and hold the small pushbutton labeled USER on the XRP's circuit board.
What you should see

The onboard LED blinks about twice a second while the button is held, and goes dark the moment you let go. That single blink is the whole round trip: the button state streams from the XRP to your laptop, your code decides, and the LED command streams back, about 50 times a second.

Step 5

Turn a wheel, read the encoders

Leave the program running and enabled, and look at the terminal where you launched robotpy sim --xrp. Every half second you get a line like L=0  R=0  gyro=0  line=0.10/0.10. Now, with the robot on the table:

  1. Slowly turn the left wheel one full turn forward by hand.
  2. Turn it back the other way.
  3. Spin the whole robot in place on the table.
  4. Slide a piece of paper with a thick dark stripe under the sensors on the XRP's nose.

An encoder works like the sensor on a bike computer: it counts small, fixed slices of a wheel turn, and counting slices tells you how far the wheel has rotated. On the XRP, one full wheel revolution is about 585 counts, the same number the capstone uses to convert counts into meters.

What you should see

L climbs by about 585 for one full forward turn of the left wheel, and counts back down when you reverse it. The right wheel moves R. Spinning the robot moves the gyro number, in degrees. The dark stripe pushes the line readings toward 1.0. Every sensor on the robot is now streaming into your code.

When it fights back

Troubleshooting

Three failures cover almost every first run:

SymptomLikely causeFix
No XRP-XXXX network in the WiFi list Dead batteries, or the robot is off or still booting Fresh batteries, power switch on, wait about 30 seconds, then refresh the WiFi list.
Sim runs but nothing prints and the LED never blinks Laptop hopped back to your usual WiFi, the --xrp flag is missing, or the robot is still Disabled Re-join XRP-XXXX, rerun robotpy sim --xrp, then set Teleop and click Enable in the GUI.
Encoder counts stay at 0 while you turn Turning the robot body instead of a wheel, or watching the wrong side Spin one wheel slowly through a full turn and watch its own number; it should move by about 585.
Going further · optional

Make the code yours. Change the blink to once a second (what number replaces the 4?). Make the LED light only when the left reflectance sensor sees dark, an instant line detector. Use setDistancePerPulse like the capstone's drivetrain does and print centimeters instead of raw counts. Or rotate the robot by hand until the gyro reads exactly 90 and check it against a corner of the table. Question to chew on: the button and the LED are half a meter of WiFi apart from the code that connects them, so what would happen to the blink if the WiFi link got slow or dropped for a second?

Checkpoint

Quick check

Pick an answer to see why.

Question 1
While the LED is blinking, where is your program actually running?
With the XRP, "deploy" means run in simulation on the laptop. robotpy sim --xrp starts your program there, and a WebSocket link over the XRP's own WiFi carries commands out and sensor readings back. Close the laptop and the robot stops.
Question 2
You hold the USER button and the LED blinks. What did that actually prove?
The button is not wired to the LED. The XRP sends the button reading to your program, teleopPeriodic computes pressed and blink_on, and the result travels back to the LED, about 50 times a second. One blink proves input, output, and your code in between.
Question 3
You turn the left wheel one full revolution by hand and L climbs by about 585. What is the encoder counting?
An encoder ticks off tiny fractions of a revolution as the wheel turns, about 585 per full turn on the XRP. Divide counts by 585 and multiply by the wheel's circumference and you get distance, which is exactly what the capstone's setDistancePerPulse conversion does.
