Lab · Part 4

Sensors,
for real.

You have read encoders, gyros, and digital sensors in lessons. In this lab you read the real ones, on the XRP, and check every number against a tape measure, a quarter turn, and a strip of tape on the floor.

Before you start

What you need

This is the same $60 robot the course capstone uses. If you already set it up for the capstone, you are two minutes from the first step.

  • An XRP kit (the robot, motors, and sensors).
  • Charged batteries for the XRP. Fresh ones matter, tired batteries make every reading in this lab drift.
  • A laptop with WPILib installed, which you set up back in Part 1.
  • A ruler or tape measure, plus a strip of dark tape and a light surface (a big sheet of white paper works).

Getting connected is the same short routine as the capstone:

  1. Install WPILib, if you have not already (Part 1).
  2. Make a new project from the XRP template or example in VS Code.
  3. Power on the XRP and connect your laptop to the XRP's WiFi network.
  4. Run the program in simulation: in VS Code, run WPILib: Simulate Robot Code. This connects to the XRP over WiFi.
  5. Open the Simulation GUI, set the robot to Teleop and click Enable when a step says go.

Remember the mental model from the capstone: your code runs in simulation on the laptop, and a WebSocket link over the XRP's own WiFi carries motor commands out and sensor readings back. The sensor numbers you are about to watch are real hardware talking to you across that link.

Follow the official guide for the exact clicks

We teach the idea here. Versions and menus change, so follow the official WPILib XRP guide for the exact, current click-by-click setup. It is kept up to date by the people who make the tools.

Step 1 · Encoders

Ticks to centimeters

An encoder is a bike computer for a robot wheel. A bike computer counts each pass of a wheel magnet and multiplies by the wheel size to get distance. The XRP's encoders tick 585 times per wheel revolution, and the wheel is 6.0 cm across, so one revolution rolls pi times 6.0, about 18.8 cm. One tick is therefore about 0.032 cm. setDistancePerPulse teaches the Encoder class that conversion once, and from then on getDistance() answers in centimeters.

Robot.java - drive straight, watch ticks and cm java
import edu.wpi.first.wpilibj.Encoder;
import edu.wpi.first.wpilibj.TimedRobot;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;
import edu.wpi.first.wpilibj.xrp.XRPMotor;

public class Robot extends TimedRobot {
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);
  private final Encoder leftEncoder = new Encoder(4, 5);
  private final Encoder rightEncoder = new Encoder(6, 7);

  // 60 mm wheels, 585 encoder counts per wheel revolution (from the XRP example)
  private static final double WHEEL_DIAMETER_CM = 6.0;
  private static final double COUNTS_PER_REV = 585.0;

  @Override
  public void robotInit() {
    double cmPerCount = (Math.PI * WHEEL_DIAMETER_CM) / COUNTS_PER_REV;
    leftEncoder.setDistancePerPulse(cmPerCount);
    rightEncoder.setDistancePerPulse(cmPerCount);
    rightMotor.setInverted(true); // the right side is mirrored
    leftEncoder.reset();
    rightEncoder.reset();
  }

  @Override
  public void robotPeriodic() {
    SmartDashboard.putNumber("Left ticks", leftEncoder.get());
    SmartDashboard.putNumber("Right ticks", rightEncoder.get());
    SmartDashboard.putNumber("Left cm", leftEncoder.getDistance());
    SmartDashboard.putNumber("Right cm", rightEncoder.getDistance());
  }

  @Override
  public void teleopPeriodic() {
    // slow and steady: drive straight while you watch the numbers
    leftMotor.set(0.4);
    rightMotor.set(0.4);
  }

  @Override
  public void teleopExit() {
    leftMotor.set(0);
    rightMotor.set(0);
  }
}
  1. Mark a start line on the floor with a small piece of tape and line the XRP's wheels up on it.
  2. Start the sim, open the SmartDashboard view in the Simulation GUI, and find the four numbers.
  3. Enable Teleop. The XRP drives forward on its own.
  4. Disable the moment Left cm reads about 50.
  5. Now measure from the start line to the wheels with the tape measure. Compare what the ruler says to what the encoder claimed.
What you should see

Ticks and centimeters climb together while the robot rolls, roughly 1550 ticks by the time the dashboard says 50 cm. Your tape measure should agree with the dashboard within a centimeter or two, coasting after you disable accounts for most of the gap. If the ruler and the dashboard disagree wildly, your distance-per-pulse math is wrong, not the ruler.

Step 2 · Gyro

Turn 90 degrees, then check the claim

Close your eyes and spin once on a swivel chair. You still know roughly how far you turned, because your inner ear senses rotation. The XRPGyro is that inner ear: it reports how many degrees the robot has rotated since the last reset(). This program spins the XRP in place and cuts the motors the instant the gyro passes 90 degrees. Depending on which way yours spins, the angle may count negative, so the code compares Math.abs of the heading, only the size matters here.

Robot.java - spin in place, stop at 90 java
import edu.wpi.first.wpilibj.TimedRobot;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;
import edu.wpi.first.wpilibj.xrp.XRPGyro;
import edu.wpi.first.wpilibj.xrp.XRPMotor;

public class Robot extends TimedRobot {
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);
  private final XRPGyro gyro = new XRPGyro();

  @Override
  public void robotInit() {
    rightMotor.setInverted(true); // the right side is mirrored
  }

  @Override
  public void teleopInit() {
    gyro.reset(); // the direction it faces right now becomes 0 degrees
  }

  @Override
  public void teleopPeriodic() {
    double heading = gyro.getAngle();
    SmartDashboard.putNumber("Heading deg", heading);
    if (Math.abs(heading) < 90.0) {
      leftMotor.set(0.4);   // left forward, right back: spin in place
      rightMotor.set(-0.4);
    } else {
      leftMotor.set(0);
      rightMotor.set(0);
    }
  }
}
  1. Stick a small tape arrow to the floor pointing the same way as the XRP's nose.
  2. Before enabling, watch Heading deg for ten seconds with the robot sitting still. Does it hold at zero or creep?
  3. Enable Teleop. The XRP spins and stops itself.
  4. Read the final heading on the dashboard, then eyeball the robot against your tape arrow. Is it really a quarter turn?
What you should see

The heading counts up (or down) fast while the robot spins, the motors cut at 90, and then the number settles a few degrees past, often 94 to 100. That overshoot is real: the robot coasts after the power cuts, and the gyro honestly reports it. Sitting still, the heading may also creep slowly by a degree or two per minute, that is gyro drift. Both are exactly the problems the Control course teaches you to beat.

Step 3 · Reflectance

Cross a tape line and watch the light

Under the XRP's nose sit two reflectance sensors. Each one shines a little light at the floor and measures how much bounces back, the way a white t-shirt bounces sunlight while a dark hoodie soaks it up. The XRPReflectanceSensor reports each side as a number from 0.0 (bright surface) to 1.0 (dark surface). Lay a strip of dark tape across white paper and you have a line the robot can see.

Robot.java - creep forward, watch both sensors java
import edu.wpi.first.wpilibj.TimedRobot;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;
import edu.wpi.first.wpilibj.xrp.XRPMotor;
import edu.wpi.first.wpilibj.xrp.XRPReflectanceSensor;

public class Robot extends TimedRobot {
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);
  private final XRPReflectanceSensor reflectance = new XRPReflectanceSensor();

  @Override
  public void robotInit() {
    rightMotor.setInverted(true); // the right side is mirrored
  }

  @Override
  public void robotPeriodic() {
    // 0.0 is a bright surface (white paper), 1.0 is dark (black tape)
    SmartDashboard.putNumber("Left reflectance", reflectance.getLeftReflectanceValue());
    SmartDashboard.putNumber("Right reflectance", reflectance.getRightReflectanceValue());
  }

  @Override
  public void teleopPeriodic() {
    // creep, so the crossing is slow enough to watch
    leftMotor.set(0.3);
    rightMotor.set(0.3);
  }

  @Override
  public void teleopExit() {
    leftMotor.set(0);
    rightMotor.set(0);
  }
}
  1. Lay the white paper down and stick a strip of dark tape across it.
  2. With the sim running but the robot disabled, hold the XRP by hand: hover the nose over the paper, then over the tape. Watch both numbers move.
  3. Set the XRP on the paper aimed at the line, a little crooked on purpose.
  4. Enable Teleop, let it creep across the tape, and disable on the far side.
What you should see

Over the white paper both values sit low, around 0.1 to 0.2. As each sensor crosses the tape its value jumps toward 0.8 to 1.0, then drops back on the far side. Because you aimed crooked, one sensor spikes before the other. That timing difference tells the robot which side of the line it is drifting toward, and it is the entire secret of line following.

When it fights back

Troubleshooting

SymptomLikely causeFix
Dashboard numbers never move Your laptop is still on the regular WiFi, so the sim never reached the XRP Join the XRP's own WiFi network, then restart the simulation
Ticks climb but the cm number is way off the tape measure setDistancePerPulse is missing or the math is off One tick should be about 0.032 cm: pi times 6.0 cm, divided by 585
Reflectance barely changes over the tape Not enough contrast between the line and the surface Use matte dark tape on white paper, and test the crossing spot by hand first
Going further · optional

Three sensors, three upgrades. Turn step 3 into a first line follower: if the left sensor sees the tape, steer left a little, if the right one sees it, steer right. Rerun step 1 with fresh batteries and again when they are tired, and compare how far the robot coasts past 50 cm each time. And in step 2, shrink the overshoot: instead of full speed until 90, slow the spin down as the heading gets close. That last one is you reinventing proportional control, which is where the Closing the Loop course begins.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Your left encoder reads 585 ticks. About how far has that wheel rolled?
The XRP encoder ticks 585 times per wheel revolution, and one revolution rolls the wheel one circumference: pi times 6.0 cm, about 18.8 cm. So each tick is about 0.032 cm. Once you hand that conversion to setDistancePerPulse, getDistance() does the multiplication for you.
Question 2
The step 2 code cuts the motors the instant the heading passes 90 degrees, but the dashboard settles at about 96. What happened?
Cutting power does not stop a spinning robot instantly, momentum carries it a few degrees further, and the gyro keeps measuring the whole time. The sensor is telling the truth about an open-loop stop. Feedback control, easing off as the target gets close, is how you shrink that overshoot, and it is what the Control course teaches.
Question 3
You drive the XRP slowly across a strip of black tape on white paper. What should the reflectance values do?
The sensor reports 0.0 for a bright surface and 1.0 for a dark one. White paper bounces the light back, so the value sits low; black tape soaks the light up, so the value jumps high while the sensor crosses it. Cross at an angle and the two sensors spike at different moments, which is the seed of line following.
