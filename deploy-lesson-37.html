real robot.

You have written command-based code, tuned it, and seen it move in simulation. Now run the same code on a small robot you can hold in your hand, no FRC build and no budget needed.

What & why

A real robot for the cost of a video game

Everything you learned in this course runs on the XRP.

The XRP (Experimental Robotics Platform) is a small, low-cost robot built on the Raspberry Pi Pico W. WPILib officially supports it, which means the same WPILib code you have been writing all course, subsystems, commands, PID, drives a real robot you can put on your desk. A full FRC robot costs thousands of dollars and fills a workshop. An XRP is roughly $35 to $70, so check the current price before you buy. It is the bridge between the simulator and a competition robot.

There is an older cousin called the Romi. The Romi is built on a Raspberry Pi instead of a Pico W, it is also WPILib-supported, and it works the same way. It is the pricier predecessor. The XRP is the newer, cheaper option, so this lesson uses the XRP, and almost everything here applies to the Romi too.

Why this matters

Code that only ever ran in a simulator can hide mistakes that real wheels and a real battery expose in seconds. Running on hardware closes the gap. The XRP makes that possible for the price of a couple of pizzas, so you can practice the whole loop, build an idea, code it, tune it, before you ever touch a competition robot.

How it works

"Deploy" here means run in simulation, over WiFi

This is the one mental model to hold on to. On a competition robot, your code is loaded onto the roboRIO and runs there. The XRP is different and simpler:

  • Your robot program runs on your laptop, in WPILib simulation.
  • The XRP creates its own WiFi network. Your laptop joins it.
  • A WebSocket link carries your motor commands out to the XRP and streams its sensor readings back, many times per second.

So when we say "deploy to the XRP," we really mean: run your code in simulation on the laptop, connected to the XRP over WiFi. Your program thinks it is talking to motors and encoders. It is, they only live across a WiFi link in a little robot on the table. The Romi uses the exact same model, so what you learn here carries straight over.

That is the whole trick: the parts of your code that matter, the subsystem, the commands, the controller, never change. Only where the motors physically live changes.

Setup

What you need, and the high-level steps

You need three things:

  • An XRP kit (the robot, motors, and sensors).
  • A laptop with WPILib installed, which you set up back in Part 1.
  • Batteries for the XRP (it runs on its own pack, not your laptop).

The setup, at a high level, is short:

  1. Install WPILib, if you have not already (Part 1).
  2. Make a new project from the XRP template or example in VS Code.
  3. Power on the XRP and connect your laptop to the XRP's WiFi network.
  4. Run the program in simulation: in VS Code, run WPILib: Simulate Robot Code. This connects to the XRP over WiFi.
  5. Open the Simulation GUI, set the robot to Teleop and click Enable, then drive it.
Follow the official guide for the exact clicks

We teach the idea here. Versions and menus change, so follow the official WPILib XRP guide for the exact, current click-by-click setup. It is kept up to date by the people who make the tools.

The hardware in code

The same command-based pattern

WPILib gives the XRP its own small set of classes, but they plug into the same patterns you already know. The XRP exposes:

  • Two drive motors: XRPMotor on channels 0 (left) and 1 (right).
  • Wheel encoders: Encoder on DIO 4 and 5 (left), 6 and 7 (right).
  • An onboard gyro: XRPGyro.
  • A servo: XRPServo, plus a rangefinder and reflectance line sensors.

Here is a Drivetrain subsystem for the XRP. Look closely: this is the same SubsystemBase shape from Part 3. Only the class names for the motors, encoders, and gyro are XRP-specific.

Drivetrain.java java
import edu.wpi.first.wpilibj.xrp.XRPMotor;
import edu.wpi.first.wpilibj.xrp.XRPGyro;
import edu.wpi.first.wpilibj.Encoder;
import edu.wpi.first.wpilibj2.command.SubsystemBase;

public class Drivetrain extends SubsystemBase {
  // XRP drive motors live on channels 0 (left) and 1 (right)
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);

  // The XRP wheel encoders are wired to these DIO channels
  private final Encoder leftEncoder = new Encoder(4, 5);
  private final Encoder rightEncoder = new Encoder(6, 7);

  private final XRPGyro gyro = new XRPGyro();

  // From the XRP example: 60 mm wheels, 585 encoder counts per wheel revolution
  private static final double WHEEL_DIAMETER_M = 0.060;
  private static final double COUNTS_PER_REV = 585.0;

  public Drivetrain() {
    double metersPerCount = (Math.PI * WHEEL_DIAMETER_M) / COUNTS_PER_REV;
    leftEncoder.setDistancePerPulse(metersPerCount);
    rightEncoder.setDistancePerPulse(metersPerCount);
    rightMotor.setInverted(true); // the right side is mirrored
    resetEncoders();
  }

  public void arcadeDrive(double speed, double rotation) {
    leftMotor.set(speed + rotation);
    rightMotor.set(speed - rotation);
  }

  public double getAverageDistanceMeters() {
    return (leftEncoder.getDistance() + rightEncoder.getDistance()) / 2.0;
  }

  public double getHeadingDegrees() { return gyro.getAngle(); }
  public void resetEncoders() { leftEncoder.reset(); rightEncoder.reset(); }
  public void resetGyro() { gyro.reset(); }
}

If that reads like the subsystems you wrote earlier, that is the point. The skills transfer because the framework is the same.

The guided project

Make the XRP drive a square

Here is your capstone challenge: drive the XRP around a square all on its own, then upgrade it with real control. We will build it in three steps.

Step 1. Two building blocks, written as commands: drive forward a set distance, and turn a set number of degrees. These are open-loop, they set a fixed speed and stop when the sensor passes a target.

Open-loop building blocks java
Command driveForward(double meters, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(speed, 0),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .beforeStarting(drivetrain::resetEncoders)
    .until(() -> drivetrain.getAverageDistanceMeters() >= meters);
}

Command turnDegrees(double degrees, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(0, speed),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .beforeStarting(drivetrain::resetGyro)
    .until(() -> Math.abs(drivetrain.getHeadingDegrees()) >= Math.abs(degrees));
}

Step 2. A square is those two blocks, four times: drive a side, turn a corner, repeat. Sequence them.

The square as a sequence java
Command driveSquare() {
  return Commands.sequence(
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5));
}

Run that and the XRP traces a square. Mostly. Each side coasts a little past the line before it stops, and on a tired battery the wheels turn slower and each side comes up short. The open-loop blocks never look at the gap between where the robot is and where you wanted it, so the errors stack up and the square does not quite close.

Step 3, the upgrade. This is where the Control course pays off. Replace the open-loop drive with a PIDController drive-to-distance, the exact idea from Closing the Loop, now running on real wheels.

Closed-loop drive, with a PIDController java
Command driveDistancePID(double meters) {
  PIDController pid = new PIDController(2.0, 0, 0);
  pid.setTolerance(0.02);
  return Commands.run(() -> {
      double out = pid.calculate(drivetrain.getAverageDistanceMeters(), meters);
      drivetrain.arcadeDrive(MathUtil.clamp(out, -0.6, 0.6), 0);
    }, drivetrain)
    .beforeStarting(() -> { drivetrain.resetEncoders(); pid.reset(); })
    .until(pid::atSetpoint)
    .andThen(() -> drivetrain.arcadeDrive(0, 0));
}

The open-loop version says "drive until distance is at least the target," then coasts. It overshoots when the battery is fresh and comes up short when the battery is weak, because it never looks at the error. The PID version watches the remaining gap and drives it toward zero, slowing down as it gets close. That is the same droop-and-correct idea from Closing the Loop, now on hardware you can hold. Want to take it further? Add a small feedforward term, or tune kP for your robot, and check the control course for how.

Going further · optional

The square is a starting line, not a finish line. Try a triangle (three sides, three 120 degree turns) and see if your turn logic still holds. Use the XRP's rangefinder to stop a fixed distance before a wall instead of after a set distance. Line-follow with the reflectance sensors, steering toward whichever side sees the line. Or take one of the mechanisms you tuned earlier, an elevator or an arm idea, and port its controller onto the XRP servo. Question to chew on: when you switched the drive from open-loop to PID, what specifically made the square close, and what would happen to each version if you swapped in a much heavier battery?

Show the thinking

The square closed because the PID drive stops at the target distance instead of past it. Open-loop picks a fixed speed and stops the instant the encoder crosses the line, so it always coasts a little further, and how far depends on speed, which depends on the battery. PID measures the gap to the target every loop and eases off as it shrinks, so it lands on the line no matter the battery. Swap in a heavier, fresher battery and the open-loop version overshoots more (faster wheels coast further), while the PID version still lands on target, it only gets there a bit quicker. That battery-proof behavior is exactly why feedback control is worth the extra few lines, and it is the whole point of the Closing the Loop course.

Quick check

Pick an answer to see why.

Question 1
When you "deploy" your code to an XRP, where does your robot program actually run?
With the XRP, your code runs in simulation on your laptop. A WebSocket link over the XRP's own WiFi network carries motor commands out and sensor readings back. "Deploy" here means run in simulation, connected to the XRP.
Question 2
In the XRP Drivetrain, which classes give you the two drive motors and the heading?
The XRP exposes its drive motors as XRPMotor (channels 0 and 1), its wheel encoders as Encoder on DIO pins, and its onboard gyro as XRPGyro. TalonFX and Pigeon are CTRE parts you would use on a full FRC robot.
Question 3
Why does the open-loop "drive until distance is at least the target" overshoot, while the PID version lands on target?
Open-loop picks a speed and stops once the encoder crosses the line, so it always coasts a bit further. PID measures the gap to the target every loop and slows down as the gap shrinks, so it settles on the line, the droop-and-correct idea from Closing the Loop.
Question 4
How much of the command-based code you wrote earlier in this course works on the XRP?
The XRP is WPILib-supported, so the structure you learned, SubsystemBase, command factories, PIDController, is the same. Only the hardware classes change to XRPMotor, Encoder, and XRPGyro. That is why the skills transfer.
Question 5
Why is the XRP a good way to practice without an FRC team's budget?
A full FRC robot costs thousands of dollars. The XRP is roughly $35 to $70 (check the current price) and runs the very same WPILib code, so you can practice the whole build-code-tune loop on real hardware on your own desk.
