Part IV · 02 Tune a Real Mechanism
The Arm

Gravity that
moves.

An elevator fights the same gravity everywhere. An arm's load changes with its angle - full at horizontal, zero straight up. One cosine fixes it.

The elevator had it easy.

Straight up and down, gravity is constant, so a single kG holds it anywhere. An arm is different: held out horizontal it strains against its full weight, but pointed straight up it barely needs anything. The gravity load follows the cosine of the angle. Cosine is just the math name for that smooth fade: picture a dial that reads 1 (full pull) when the arm is flat, slides down as the arm rises, and reaches 0 (no pull) when it points straight up. You never work it out by hand, Phoenix and WPILib compute it for you:

hold voltage = kG · cos(angle) angle measured from horizontal · cos(0°) = 1 (full) · cos(90°) = 0 (none)

So feedforward for an arm needs to know the angle, and scale kG by its cosine. Get that right and the arm holds rock-steady at any position; get it wrong and it sags or drifts. Let's see both.

Try it

Cosine on, cosine off

Drag the target angle around. With cosine compensation on, the arm holds wherever you point it. Switch it off - now feedforward applies a flat kG that's only right at horizontal, and the arm drifts high as you raise the target.

Predict first
You switch cosine compensation off, so feedforward applies a flat kG set for horizontal. Then you raise the target angle up toward vertical. What happens to the arm?
Gravity's pull on an arm fades with the cosine of the angle, full at horizontal and zero straight up. A flat kG is only correct at horizontal, so as the arm rises the real load shrinks while feedforward keeps pushing the full horizontal amount, overdriving it past the target. Cosine compensation scales kG by cos(angle) to track that fade. Run it with cosine off and raise the target to watch the arm drift high.

In code

One line tells the motor about gravity

On Phoenix 6, you don't compute the cosine yourself - you set the gravity type, and the device scales kG by cos(position) on every loop. The one rule: the arm's zero must be horizontal, so cos(0) = 1 lines up with full gravity.

Arm gains - the key difference Java · Phoenix 6
cfg.Slot0.kG = 0.40;                                  // volts to hold the arm HORIZONTAL
cfg.Slot0.kS = 0.10;  cfg.Slot0.kV = 0.9;  cfg.Slot0.kA = 0.02;
cfg.Slot0.kP = 60;    cfg.Slot0.kD = 0.5;
cfg.Slot0.GravityType = GravityTypeValue.Arm_Cosine;     // ← scales kG by cos(angle)

// position 0 = horizontal. Use a FusedCANcoder so the arm knows
// its true angle the instant the robot powers on (Lesson III.2).

That's the whole change from the elevator: Arm_Cosine instead of Elevator_Static, and a horizontal zero. Everything else - Motion Magic, the control request, the workflow - is identical. On the WPILib side it's ArmFeedforward, which takes the angle directly:

If you compute feedforward yourself Java · WPILib
ArmFeedforward ff = new ArmFeedforward(kS, kG, kV, kA);
double volts = ff.calculate(angleRadians, velocityRps);  // kG·cos(angle), for you

Going further · optional

The cosine trick assumes the heavy part of the arm sits at the far end, so the pull is strongest at horizontal. But put a real game piece in the gripper and the arm's effective weight, and the angle where its mass actually balances, both shift, which means the kG you measured empty is now slightly off. Top teams handle this by switching to a second gains slot, or scaling kG, the moment they pick up a piece. Worth chewing on: if your kG is tuned empty, would a loaded arm tend to sag below the target or drift above it, and at which angle would you notice it most?

Show the thinking

It sags below the target. A game piece adds weight the empty kG never accounted for, so the feedforward holds up less than gravity now pulls down, and the small kP cannot make up the rest, leaving the arm low. You notice it most near horizontal, because the gravity torque follows the cosine of the angle and peaks there, so the shortfall is largest. Near vertical the cosine term fades toward zero and the sag nearly disappears, which is the clue to bump kG, or switch to a loaded gains slot, the moment a piece is in hand.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Why does an arm's gravity feedforward depend on its angle?
Horizontal, the arm's whole weight pulls down on the pivot (cos 0° = 1). Straight up, the weight pulls along the arm, creating no turning torque (cos 90° = 0). So the hold voltage is kG·cos(angle).
Question 2
On Phoenix 6, how do you get the cosine scaling?
GravityType.Arm_Cosine tells the onboard loop to scale kG by cos(position). Because it uses the measured position, zero must be horizontal so cos(0)=1 corresponds to full gravity.
Question 3
Which sensor setup helps an arm know its true angle the moment the robot turns on?
The rotor only counts from boot, so it can't know an absolute angle. A FusedCANcoder gives the true angle immediately - essential for an arm, where cosine feedforward needs a correct angle from the first instant.
