Question 1
Each swerve module has a steer motor and a drive motor. Which Part IV mechanism does each one behave like?
A swerve module reuses Part IV directly: the steer loop points the wheel to an angle (position control, like the arm), and the drive loop holds a wheel speed (velocity control, like the flywheel). Four modules means eight motors, all built from mechanisms you already know.
Question 2
Each steer motor uses a FusedCANcoder. Why does a position loop like steering need an absolute sensor at power-on, when drive (velocity) does not?
Steering is position control, so it needs the wheel's real angle the moment the robot powers on, or it starts pointed wrong. The FusedCANcoder gives an absolute angle while still allowing the crisp fast loop. A velocity loop tracks rate of change, so it does not need an absolute starting reference.
Question 3
A camera reports tx, the horizontal angle from the robot's bore-sight to a target. You auto-aim by spinning the robot with omega = -kP times tx. Which earlier idea is this, and what is its known weakness?
Setpoint is tx = 0 (centered), so omega = -kP times tx is pure proportional control from Lesson 2, now reading a camera. Tracking a moving target leaves a small steady offset, since P needs an error to keep turning. Adding an I term or target-velocity feedforward fixes it.
Question 4
Odometry alone drifts as wheels slip. The team blends in AprilTag pose estimates with addVisionMeasurement(). What makes vision worth fusing in, and what is the main complication?
A known tag gives an absolute field location that corrects odometry drift. The catch is latency: frames arrive delayed, so you time-stamp them, weight them against odometry, and reject wild jumps rather than trusting vision blindly.
Question 5
The state-space lesson reveals that the PD control you used all course is a special case. On a system with state [position, velocity], state feedback u = K times (reference minus state) turns out to be which two gains, and what does an LQR add?
On a position-velocity system, the state gains are exactly kP (on position error) and kD (on velocity error), so PD is state feedback. An LQR picks K optimally from Q (penalize error, tighter) and R (penalize effort, gentler), which scales to multi-state systems where eyeballing gains is hopeless.
