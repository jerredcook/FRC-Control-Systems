Swerve Drive in Code

Every wheel
steers.

Swerve gives each of four wheels its own steering motor. Point them together and the robot slides any direction; point them in a pinwheel and it spins. The math that aims them is swerve kinematics.

Four modules, each pointing wherever it needs to.

A swerve module is a drive wheel plus a steering motor, so it can face any angle and roll at any speed. With four of them the robot is holonomic: it can translate any direction and rotate at the same time. The trick is figuring out each module's angle and speed from one chassis command. Drag the translation pad and turn the rotation, and watch the four arrows.

Predict first
Before you run it: you set pure rotation with the omega slider and no translation. How do the four module arrows point?
To spin in place, each module must push along the circle around the robot's center, so the four arrows form a pinwheel, each one perpendicular to its line from center. Pure translation instead points all four the same way. Run it, set omega with no translation, and watch the pinwheel form, then add translation to see them swing.

🎯 Missions clear them with the sim above
○Drag translation at least halfway out with omega at 0. All four arrows point the same way, so the robot slides without turning.
○Press Center, then slide omega to 0.5 or more with no translation. The arrows form a pinwheel, each wheel pushing along the circle around the center.
○Combine them: drag translation to the edge and push omega to 1.0. The fastest module pins at 100% and every arrow shrinks together - that is desaturateWheelSpeeds keeping the robot on course. The longest arrow is the wheel whose spin push lines up with your travel direction.
★Gold: with omega near 0.5, drag translation until one wheel's arrow vanishes while the other three keep pushing. You have put the turn center right on that wheel, so the robot pivots around it, like planting one foot and swinging around it.
In code

Kinematics does the trig

You tell WPILib where the modules sit, then hand it a chassis command. toSwerveModuleStates returns one target angle and speed per module:

swerve_subsystem.py Python · RobotPy
# where each module sits relative to robot center (meters)
kinematics = SwerveDrive4Kinematics(
    fl_location, fr_location, bl_location, br_location)

# the chassis motion you want (field-relative uses the gyro from 4.3)
speeds = ChassisSpeeds.fromFieldRelativeSpeeds(
    vx, vy, omega, self.gyro.getRotation2d())

# one angle + speed per module
states = kinematics.toSwerveModuleStates(speeds)
states = SwerveDrive4Kinematics.desaturateWheelSpeeds(states, MAX_SPEED)

for i in range(4):
    self.modules[i].set_desired_state(states[i])
Field-relative is the swerve superpower

Because each wheel steers, swerve can drive "north" while the robot body points any direction. fromFieldRelativeSpeeds rotates your stick command by the gyro heading, so push-away always means down-field, just like Lesson 4.3 promised. desaturateWheelSpeeds then scales every module down together if one would exceed the max, so the robot keeps its commanded direction instead of veering.

Going further · optional

A steering motor never needs to spin more than 90 degrees to reach any target: if a wheel is asked to point "backward," it is faster to flip the wheel 180 and run the drive motor in reverse instead. WPILib's SwerveModuleState.optimize() does exactly this trick. Try reasoning it out: if a module currently points at 10 degrees and you command 170 degrees, is it shorter to rotate the steering 160 degrees forward, or to rotate it 20 degrees the other way and drive backward?

Show the thinking

The 20 degree move is shorter, and it is what optimize() picks. Pointing at 170 degrees and pointing at minus 10 degrees push the wheel along the same line, only driven in opposite directions, so the module can reach minus 10 degrees (a 20 degree turn from 10) and run the drive motor backward to get the same motion. That beats swinging the steering a full 160 degrees, and because no flip is ever worse than 90 degrees of steering, the wheel snaps to its heading and starts pushing sooner.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What can a swerve drive do that a tank drive can't?
Independent steering on every module makes swerve holonomic: it can slide sideways, drive diagonally, and spin all at once. Tank can only move along the direction it's pointing.
Question 2
You command pure rotation (omega only, no translation). How do the modules point?
For rotation, each module must push perpendicular to its line from the center, forming a pinwheel. That's exactly what the kinematics computes from omega.
Question 3
What does kinematics.toSwerveModuleStates(speeds) return?
It converts one chassis command (vx, vy, omega) into four SwerveModuleStates, each carrying the angle to steer to and the speed to drive. The module controllers then chase those targets.
