Closing the Loop on Vision

Aim with the
camera.

You have tx, the angle to the target. That's an error. A PID loop is a feedback recipe that pushes harder the bigger the error and eases off as it shrinks (taught in full in the Control course, Part 5): feed it tx and turn until tx is zero. Vision aiming is that same loop with a camera for a sensor.

tx is the error. The setpoint is zero.

Point the robot at an AprilTag automatically: each loop, read tx, run it through a PID controller whose target is 0, and apply the output as a turn. The robot spins until it's dead-on. It's the same loop you tuned in the Control course, only the camera supplies the measurement. Set an offset, press Aim, and try different gains.

A moderate kp slides the robot smoothly onto the tag and locks. Crank kp up and, with the camera's small delay, it overshoots and hunts back and forth, the exact same over-tuning you saw with mechanisms. The trace at the bottom is tx over time.

In code

It really is just a PIDController

Make a controller with the setpoint at 0, feed it tx every loop, and apply the result as your turn command:

aim_command.py Python · WPILib
from wpimath.controller import PIDController

aim = PIDController(0.03, 0, 0.001)

def execute(self):
    if not self.camera.has_target():
        self.drive.stop()
        return
    tx   = self.camera.get_yaw()       # error from center
    turn = self.aim.calculate(tx, 0)   # drive tx toward 0
    self.drive.arcadeDrive(0, turn)      # turn in place

def isFinished(self):
    return abs(self.camera.get_yaw()) < 1.0   # locked on
Latency is the new twist

A camera reports a few frames late, so the robot acts on slightly stale tx. With gentle gains that's fine, but a high kp fights that delay and the robot oscillates, the same instability the Control course warned about, now from sensor lag instead of too much gain alone. The fixes are the same: a sane kp, a touch of D, a tolerance band, and only shooting once isFinished() says you're locked.

Going further · optional

Try this: add aim.setTolerance(1.0) when you build the controller, then let isFinished() return aim.atSetpoint() instead of comparing tx yourself, so the lock test and the controller share one number. Then think about the harder case: if the tag is to your right but a wall is in the way, a pure aim loop spins forever. How would you give up gracefully, for example a timeout, so the robot is never stuck mid-match?

Show the thinking

Put a hard cap on how long the command is allowed to run, and end it when the cap is hit even if it never reached the setpoint. The clean way is to wrap the aim command with withTimeout(1.5), or check a Timer inside isFinished(), so the command always finishes one way or another. Have it report whether it gave up or truly locked on, so the next step can react, for example skip the shot or back up and try a new angle. A loop that can only end on success is a loop that can hang the whole match.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
To aim the robot at a tag, what do you use as the PID error?
tx is how far off-center the target is. Set the controller's target to 0 and it computes a turn that drives tx to zero, pointing the robot straight at the tag.
Question 2
Why might a high kp make vision aiming oscillate?
Camera latency means the robot corrects based on stale tx. A large gain overshoots before the new measurement catches up, so it swings back and forth. Lower kp or add a little D.
Question 3
Before the robot shoots, what should isFinished() check?
Gate the shot on being aimed: only finish (and fire) once tx is within a tolerance band. Shooting mid-swing sends the game piece wide.
