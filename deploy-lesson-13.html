Commands

Actions over
time.

A command is a verb that plays out across many loops. Four little methods describe its whole life - start, repeat, finish, clean up.

A command has a lifecycle.

When the scheduler runs a command, it calls four methods at the right moments: initialize once at the start, execute every loop while it runs, isFinished every loop to ask "done yet?", and end once when it stops. Step a "run intake until we have a game piece" command through its life:

execute() and isFinished() repeat every loop until isFinished() returns true - then end() runs once and the command is done.

In code

The four methods - and the easy way

Written out as a full class, those four methods look like this. Notice addRequirements - it tells the scheduler which subsystem this command needs (Lesson 3.6):

A Command class Java · WPILib
public class RunIntakeUntilLoaded extends Command {
  private final Intake intake;
  public RunIntakeUntilLoaded(Intake intake) {
    this.intake = intake;
    addRequirements(intake);             // I need the Intake subsystem
  }
  @Override public void initialize() { }
  @Override public void execute()    { intake.run(); }
  @Override public boolean isFinished() { return intake.hasGamePiece(); }
  @Override public void end(boolean interrupted) { intake.stop(); }
}

But you'll rarely write all that. WPILib gives subsystems command factories that build the same thing inline - far less code for simple actions:

The same command, inline Java · WPILib
intake.run(intake::run)                 // execute(): run the intake every loop
      .until(intake::hasGamePiece);    // isFinished(): stop when loaded
end() always runs - even when interrupted

If another command grabs the Intake mid-run, yours is interrupted: its end(true) still fires so it can stop the motor cleanly. That guarantee is why you put "shut things off safely" in end() - it can't be skipped.

Going further · optional

Here is a gotcha that bites teams: an inline factory like runOnce(...) or run(...) built directly on a subsystem usually adds that subsystem as a requirement for you, but a bare Commands.run(() -> ...) with no subsystem requires nothing, so two of them can quietly fight over the same motor. Try writing the same simple action two ways and ask: which one did the scheduler protect, and how would you confirm a command's requirements before trusting it?

Show the thinking

The scheduler protected the one built on the subsystem, like subsystem.run(...), because that factory adds the subsystem as a requirement, so the scheduler knows to interrupt any other command that needs it. The bare Commands.run(() -> ...) declares no requirement, so the scheduler sees no conflict and lets two of them run at once, both writing to the same motor with the last write winning. To confirm a command's requirements before trusting it, call command.getRequirements() and check the subsystem is in the set, or pass the subsystem explicitly as the second argument to Commands.run(...) so the requirement is never left out by accident.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Which command method runs every single loop while the command is active?
execute() is called every loop while the command runs. initialize() and end() each run exactly once - at the start and the stop.
Question 2
What makes a command finish on its own?
Every loop the scheduler checks isFinished(). The moment it returns true, the command's end() runs and it's removed. Return false forever and it runs until interrupted.
Question 3
A command is interrupted partway through. What happens to end()?
end() always runs - with interrupted = true when something cut it short. That's why "turn things off" belongs in end(): it's guaranteed to fire either way.
