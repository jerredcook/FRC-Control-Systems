Debugging & Robust Code

Survive the
match.

Tests catch logic bugs on your laptop. The field throws physics at you instead: a battery that sags, a sensor that drops out, a command that fires at the wrong moment. Robust code expects all of it and keeps running.

The classic field killer: the brownout.

Push too many mechanisms at once and the battery voltage sags. Drop below about 6.8 volts and the roboRIO cuts power to motors to save itself, mid-match, with no warning. Current limits cap how much each motor can pull so the bus never crashes. Crank the load up below and watch the battery, then turn limits on and try again.

Predict first

In code

Cap the draw, guard the inputs

Two habits: limit current at the source, and never trust data you haven't checked. A few lines of defense keep the robot alive when the field gets ugly:

Robust configuration Python · Phoenix 6
cfg = TalonFXConfiguration()
cfg.current_limits.supply_current_limit = 40            # amps pulled from the battery
cfg.current_limits.supply_current_limit_enable = True
motor.configurator.apply(cfg)

# guard inputs instead of crashing on bad data
if camera.has_target():
    aim(camera.get_yaw())                            # don't read a target that isn't there

# shed load if the battery is getting dangerously low
if RobotController.getBatteryVoltage() < 7.0:
    non_essential.stop()                              # keep the drivetrain alive
Defensive code expects the worst

Beyond current limits: a SlewRateLimiter stops instant full-throttle spikes that yank the voltage down; checking has_target() before reading a camera avoids acting on stale data; and giving every command a safe default means a dropped sensor coasts to a stop instead of running away. The bench is forgiving. The field, with a tired battery and a real opponent, is not, so assume every input can go wrong and handle it.

Going further · optional

There are two current limits worth knowing apart: the supply limit (amps pulled from the battery, what protects against brownout) and the stator limit (amps through the motor itself, what protects against burning it up when a mechanism stalls). Try reasoning about why a jammed intake can be fine on the battery yet still cook a motor. And here is the deeper habit: a brownout is not a freak event, it is a budget problem. Could you add up the worst-case draw of every mechanism running at once and check it fits the battery before you ever drive?

Show the thinking

A jammed intake can sit at a low command, so the controller only pulls a modest amount from the battery (low supply current) while pushing a huge stalled current through the motor windings (high stator current). The battery is fine because little is leaving it, but the motor heats up fast because nearly all that current turns into heat in a coil that is not spinning. So you want a stator limit to save the motor and a supply limit to save the battery, and they are not the same number. The budget idea works: add up the expected draw of drivetrain plus every mechanism at once, leave headroom, and a brownout stops being a surprise and becomes something you designed out.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What is a brownout?
Under heavy load the voltage drops; below about 6.8 V the roboRIO disables motor outputs to keep itself alive. It happens mid-match with no warning, which is why you prevent it.
Question 2
How do current limits help prevent brownouts?
Voltage sag is driven by current. Capping each motor's supply current keeps the combined draw in a safe range, so the bus voltage stays above the brownout threshold even under load.
Question 3
Which is an example of defensive coding?
Guarding inputs, checking has_target() before using get_yaw(), means you never act on data that isn't there. Safe defaults and input checks keep a dropped sensor from turning into a runaway robot.
