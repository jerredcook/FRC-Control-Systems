Inheritance & Interfaces

Build on what
exists.

A class can extend another, inheriting everything it has and customizing the parts it wants. It's the reason every WPILib subsystem starts with extends SubsystemBase.

You rarely start a robot class from nothing.

WPILib gives you a base class - SubsystemBase - that already knows how to register with the scheduler and run every loop. When you write class Elevator extends SubsystemBase, your Elevator inherits all of that for free. You then override the one method you care about (periodic) and add your own. In code

extends and @Override

Two keywords do all the work. extends says "build on this parent." The @Override annotation marks a method as replacing a parent's - and the compiler checks you spelled it right, catching a whole class of silent bugs:

Elevator.java Java · WPILib
public class Elevator extends SubsystemBase {   // inherit everything SubsystemBase has
  private final TalonFX motor = new TalonFX(10);   // new field

  @Override
  public void periodic() {        // replace the empty parent periodic()
    // your control code runs here, every loop
  }

  public void setGoal(double m) { /* your own method */ }
}
Interfaces - a promise instead of a parent

An interface is a list of methods a class promises to provide - a contract, not a parent. WPILib's Command, Sendable, and others are interfaces. You'll mostly use them rather than write them; just know "implements X" means "this class fills out X's required methods."

Going further · optional

Try removing the @Override line above a method whose name you then misspell, like periodc(), and see what the compiler does. With @Override it errors, because no parent method matches; without it, Java happily makes a brand-new method that the scheduler never calls, so your code "runs" but does nothing. That silent failure is exactly the bug @Override exists to catch. Worth pondering: why is an error at compile time so much kinder than one discovered on the field?

Show the thinking

A compile error stops the build right at your desk, names the file and line, and costs you a quick rename before anything ships. The same mistake found on the field shows up as a robot that behaves wrong with no obvious cause, under match pressure, where you cannot easily attach a debugger or read the code calmly. @Override turns a silent "this method is never called" bug into a loud, early, located one, which is the whole point: catch it when it is cheap, not when it is costing you a match.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
When Elevator extends SubsystemBase, what does it get?
Inheritance gives the child everything the parent has. Elevator automatically registers with the scheduler and gets called each loop - you only write the parts that are unique to your elevator.
Question 2
What does @Override do?
@Override says "I'm replacing an inherited method." If you misspell the name or get the parameters wrong, the compiler errors instead of silently creating a new, never-called method - a classic bug it prevents.
Question 3
How is an interface different from a parent class?
Extending a class inherits its actual code. Implementing an interface only promises to supply the methods the interface lists - the contract. WPILib uses both; you mostly consume interfaces like Command and Sendable.
