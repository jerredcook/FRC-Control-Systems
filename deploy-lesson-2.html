Setting Up WPILib

Your first
project.

Install the tools, generate a project, and meet every file it makes for you - so the folder full of code stops looking like a maze.

You don't install "Java" and start typing.

FRC programming runs on WPILib - a library plus a customized copy of VS Code with the whole toolchain baked in. One installer sets it all up, and a menu command stamps out a complete starter project. Three steps and you're writing robot code:

1

Install WPILib

Download the WPILib release installer for your season and run it. It gives you a dedicated WPILib VS Code with Java, Gradle, and the robot tools already configured - separate from any other VS Code you have.

2

Create a project

Open the WPILib VS Code, hit Ctrl/Cmd + Shift + P, run WPILib: Create a new project. Pick the Command Robot template, language Java, and type your team number.

3

Open it

That's it - you have a full, buildable robot project. No blank files, nothing missing. Now let's see what's inside.

You'll spend ~95% of your time in subsystems/, commands/, and RobotContainer.java. The rest is scaffolding that just works.

Two flavors

Command Robot vs Timed Robot

When you create the project you choose a template. Timed Robot is the bare-bones version - just the Robot.java methods from the last lesson. Command Robot adds the command-based structure: a RobotContainer, and folders for subsystems and commands.

Pick Command Robot

Nearly every successful FRC team uses command-based - it keeps a growing robot organized instead of a tangle of if-statements. This whole course teaches it, starting in Part 3. If a menu offers "Command Robot Skeleton," that's the clean starting point.

Going further · optional

Try this: open the generated build.gradle and find the line that sets def includeDesktopSupport. Flipping it to true lets you run the robot code in simulation on your laptop with no roboRIO attached, which is how you can test logic at home. Skim the comments in that file and see how much of the build you can follow.

Show the thinking

Setting def includeDesktopSupport = true tells Gradle to also build the desktop versions of the WPILib libraries, so you can run simulateJava and exercise your logic without a roboRIO. The cost is that desktop builds are not what the robot runs, so a few hardware-only pieces will not behave in sim, and you still deploy the normal build to the robot. The wider lesson of build.gradle is that the project is not magic: it lists the WPILib and vendor dependencies, the deploy target, and the tasks (build, deploy, simulate) the toolchain runs for you.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
How do you start a new robot project?
The WPILib VS Code generates a complete, buildable project from a template - pick Command Robot, Java, and your team number. You never start from a blank file.
Question 2
Which files will you actually edit most of the time?
Your robot's behavior lives in subsystems, commands, and the wiring in RobotContainer. Main.java, gradlew, and build.gradle are scaffolding you rarely touch.
Question 3
Why choose the Command Robot template over Timed Robot?
Command-based gives you subsystems and commands - a clean way to scale from one mechanism to a whole robot without your code turning into spaghetti. It's what this course (and most teams) use.
