Meet Phoenix 6 & the Kraken

The motor that
thinks.

A Kraken isn't just a motor - it's a controller with its own processor, running your control loop on board at 1 kHz. Time to move the loop off the roboRIO.

Your loop has been running in the wrong place.

In Part II, your PID lived on the roboRIO and ran every 20 ms - 50 times a second. That's fine, but the roboRIO is busy with everything else on the robot, and 50 Hz is slow. Inside a Kraken X60 (and a Falcon 500) is a TalonFX motor controller with its own processor that can run the very same loop at 1,000 Hz - 20× faster - right next to the motor it's driving.

CTRE's software for these devices is Phoenix 6, and the desktop app for configuring and watching them is Tuner X. The big idea of this whole part: stop sending the motor raw voltage, and start sending it a target - let it close the loop itself.

See the difference

Two places the loop can live

Toggle where the control loop runs. On the roboRIO, your code computes volts and ships them out 50 times a second. On the TalonFX, you send a target once and the motor grinds out its own loop 1,000 times a second - watch the iteration counters pull apart.

Predict first
You switch the loop to run on the TalonFX instead of the roboRIO and let the same number of seconds pass. How do the two iteration counters compare?
The roboRIO loop runs every 20 ms, about 50 times a second. The TalonFX runs its loop onboard at 1,000 times a second, roughly 20 times more often, so in the same seconds it racks up about 20 times the iterations. More corrections per second means it catches a disturbance sooner. Run the sim and watch the two counters pull apart.

Why bother

Faster, steadier, off your plate

Running the loop on the motor isn't just bragging rights. A 1 kHz loop reacts to a stalled mechanism or a bumped arm almost instantly, where a 50 Hz loop might let it sag for a whole frame. The timing is rock-steady - no competing with vision code for CPU. And your roboRIO is freed up to think about strategy instead of grinding PID for eight motors at once.

The same ideas, a new home

Nothing you learned changes. It's still measure → compare → act, still kP/kI/kD plus kS/kG/kV/kA, still motion profiles. You're just configuring the motor to run that loop itself, and sending it targets instead of voltages. Every concept from Parts I and II carries straight over.

The cast

The pieces you'll work with

TalonFX

The controller inside every Kraken and Falcon. It drives the motor and runs the onboard loop. You talk to it in code by its CAN id.

CAN bus

The single wire-pair network linking every device - motors, CANcoders, the Pigeon gyro - to the roboRIO. Each device has a unique id.

Phoenix 6

CTRE's Java/C++/Python library. It's how your code configures devices, sends control requests, and reads sensor signals.

Tuner X

The desktop app to update firmware, set CAN ids, plot signals live, and run SysId - your dashboard for the whole bus.

In code, grabbing a motor and reading its position is two lines. Phoenix keeps that position fresh in the background as a status signal, already in rotations:

Talk to a Kraken Java · Phoenix 6
TalonFX motor = new TalonFX(10);              // CAN id 10 on the default (rio) bus
// on a CANivore bus instead:  new TalonFX(10, "canivore")

double rotations = motor.getPosition().getValueAsDouble();  // a status signal, auto-refreshed
double rps       = motor.getVelocity().getValueAsDouble();  // rotations per second

That's the foundation. Next lesson we configure this motor - current limits, sensor ratios, brake mode - so it behaves the way your mechanism needs.

Going further · optional

The motor runs its loop at 1 kHz, but a status signal traveling back over the CAN bus only updates as fast as that wire allows, by default around 50 to 100 times a second, like a fast worker whose progress reports you only read once in a while. So the onboard loop is genuinely quick even though the number your roboRIO sees refreshes slower. For tighter timing, teams raise the update rate or move devices onto a faster CANivore bus. Worth pondering: why is it fine for the control to be faster than what you can watch?

Show the thinking

Because the onboard loop does not wait on the number you see. It runs on the motor's own local measurement at full 1 kHz, so it reacts instantly to what the mechanism is actually doing. The slower status frame is only there for your roboRIO to log, display, or make higher level decisions, and those jobs are fine with a 50 to 100 Hz snapshot. The control quality comes from the fast local loop, not from how often you read it back.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What is a TalonFX?
The TalonFX is the brain built into each Kraken/Falcon. It drives the motor and can run PID + feedforward itself at 1 kHz - you address it in code by its CAN id.
Question 2
Why is running the loop on the TalonFX (1 kHz) better than on the roboRIO (50 Hz)?
A 1 kHz loop catches disturbances almost instantly and isn't fighting the rest of your code for CPU time. The roboRIO just sends targets and reads status, leaving it free for higher-level work.
Question 3
In the on-motor approach, what does your roboRIO code send to the TalonFX?
You send a control request with a target. The TalonFX runs PID + feedforward against that target on board, so you're commanding "go here" instead of computing volts yourself.
