Inheritance & Interfaces

Build on what
exists.

A class can build on another, inheriting everything it has and customizing the parts it wants. It's the reason every WPILib subsystem starts with class Elevator(Subsystem).

You rarely start a robot class from nothing.

WPILib gives you a base class - Subsystem - that already knows how to register with the scheduler and run every loop. When you write class Elevator(Subsystem), your Elevator inherits all of that for free. You then override the one method you care about (periodic) and add your own. In code

Subclassing and overriding

Putting the parent in parentheses does all the work. class Elevator(Subsystem) says "build on this parent." To override a method, you just define one with the same name - your version replaces the parent's. Calling super().__init__() runs the parent's constructor so its setup still happens:

elevator.py Python · WPILib
class Elevator(commands2.Subsystem):    # inherit everything Subsystem has
    def __init__(self):
        super().__init__()                # run the parent's setup first
        self.motor = TalonFX(10)         # new attribute

    def periodic(self):                  # replace the empty parent periodic()
        # your control code runs here, every loop
        pass

    def set_goal(self, m: float):       # your own method
        pass
Interfaces - a promise instead of a parent

An interface is a list of methods a class promises to provide - a contract, not a parent. Python often handles this with duck typing: if your object has the right methods, it just works, no formal declaration needed. When you want the contract written down, an abstract base class (from abc import ABC, abstractmethod) lists the methods a subclass must fill in. WPILib's Command and Sendable play this role; you'll mostly use them rather than write them.

Going further · optional

Try misspelling the overridden method, like periodc(), and see what happens. Python will not warn you, because to it that is a brand-new method that nothing calls, so your code "runs" but the scheduler never fires your control loop. That silent failure is sneaky. One guard against it: add the @override decorator (from typing), which lets a type checker like mypy flag a name that matches no parent method. Worth pondering: why is catching that mistake before you deploy so much kinder than discovering it on the field?

Quick check

Question 1
When Elevator is written as class Elevator(Subsystem), what does it get?
Inheritance gives the child everything the parent has. Elevator automatically registers with the scheduler and gets called each loop - you only write the parts that are unique to your elevator.
Question 2
How do you override an inherited method in Python?
Defining a method with the same name as a parent's replaces it for your class. There's no required keyword - though a @override decorator lets a type checker confirm the name really matches a parent method, catching a misspelling.
Question 3
How is an interface different from a parent class?
Inheriting from a class gets its actual code. An interface (an abstract base class, or just duck typing) only requires that you supply the methods it lists - the contract. WPILib uses both; you mostly consume contracts like Command and Sendable.
