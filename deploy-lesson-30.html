Pose Estimation

Erase the
drift.

Odometry is smooth but slowly wrong. Vision is exact but occasional and jumpy. Blend them and you get the best of both: a pose that moves smoothly and never wanders, because each AprilTag sighting pulls it back to the truth.

Two imperfect sources, one good answer.

A pose estimator runs odometry every loop for smooth motion, then folds in a vision measurement whenever the camera sees an AprilTag. Because tags sit at known field spots, that measurement is absolute, it corrects the accumulated drift instead of adding to it. Drive around with vision off and watch it wander; turn vision on and watch it snap straight.

In code

update() always, addVisionMeasurement() sometimes

The estimator looks just like odometry, plus one extra call. Run update() every loop for the wheels and gyro; call addVisionMeasurement() only when the camera has a fix:

DriveSubsystem Java · WPILib
SwerveDrivePoseEstimator estimator = new SwerveDrivePoseEstimator(
    kinematics, gyro.getRotation2d(), modulePositions, new Pose2d());

@Override public void periodic() {
  estimator.update(gyro.getRotation2d(), modulePositions);   // odometry, every loop

  var visionEst = vision.getEstimatedPose();
  if (visionEst.isPresent()) {                            // only when a tag is seen
    estimator.addVisionMeasurement(
        visionEst.get().pose, visionEst.get().timestampSeconds);
  }
}

Pose2d pose = estimator.getEstimatedPosition();
Trust weights decide who wins

The estimator doesn't blindly jump to the camera; it blends, weighted by how much you trust each source. You give each one a "trust dial" (in code these are called standard deviations, where a smaller number means "trust this more"). Trust vision too much and the pose twitches on every noisy frame; too little and drift creeps back. Tuned well, you get odometry's smoothness between sightings and vision's accuracy at each one. This is the fix for the drift you watched accumulate in Lesson 6.3.

Going further · optional

The trust dials (standard deviations) do not have to be fixed. A real gotcha that bites teams: a tag seen from far away, or at a steep angle, gives a much shakier pose than one seen up close and head-on. Try scaling your vision trust by distance or by how many tags are in view, trusting a close two-tag fix a lot and a single faraway tag a little. What would go wrong if you fed in a pose from a tag the camera only half-saw at the edge of the frame?

Show the thinking

A tag cut off or smeared at the edge of the frame gives a corner solve that can be badly wrong, so the pose it reports may jump several feet from where the robot really is. If you feed that into the estimator with normal trust, it yanks your blended pose toward the bad reading, and the whole match your odometry fights a lie. The fix is to reject suspect frames before they go in: skip tags below a size or ambiguity threshold, and widen the standard deviations so a shaky fix barely nudges the estimate. Trash in with high trust is worse than no vision at all.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does a pose estimator blend together?
It runs odometry every loop for smoothness and folds in vision fixes when a tag is seen. Together they give a pose that's both stable between sightings and accurate at each one.
Question 2
Why is a vision measurement "absolute" while odometry drifts?
Each tag's location is published, so seeing one anchors the robot to a known place. Odometry only adds up motion, so its errors accumulate with nothing to correct them, until vision does.
Question 3
Which call injects a camera fix into the estimate?
update() handles odometry every loop; addVisionMeasurement() is the extra call you make only when the camera has a tag, blending that absolute fix into the running estimate.
