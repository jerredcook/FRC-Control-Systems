NetworkTables & Dashboards

See what the
robot sees.

NetworkTables is a shared notebook both the robot and your laptop can read and write. Publish a number and it appears on a dashboard instantly. Change a number on the dashboard and the robot reads it back.

A live, two-way table between robot and driver station.

Every loop, your robot writes values into NetworkTables by name: "Shooter/RPM", "Intake/hasNote". A dashboard like Elastic, Shuffleboard, or AdvantageScope subscribes and draws them. It runs the other way too: the dashboard can write a value, and the robot reads it, which is how you tune a number without redeploying. Watch the robot publish, then drag the tuning slider.

In code

Publish out, read back

The quick way is SmartDashboard: one call per value, per loop. For high-rate logging, publish a typed signal once and set it each loop. Both land in the same NetworkTables:

Telemetry Python · RobotPy
from wpilib import SmartDashboard
from ntcore import NetworkTableInstance

# publish telemetry every loop - shows up on the dashboard by name
SmartDashboard.putNumber("Shooter/RPM", self.shooter.get_rpm())
SmartDashboard.putBoolean("Intake/hasNote", self.intake.has_game_piece())
SmartDashboard.putNumber("Arm/angleDeg", self.arm.get_angle())

# read a value the dashboard sends back - live tuning, no redeploy
k_p = SmartDashboard.getNumber("Shooter/kP", 0.18)

# modern, faster: publish a typed signal once, set it each loop
self.rpm_pub = NetworkTableInstance.getDefault() \
    .getTable("Shooter").getDoubleTopic("RPM").publish()
self.rpm_pub.set(self.shooter.get_rpm())
You can't debug what you can't see

Robots fail in ways you can't watch from the stands. Logging a few key signals, RPM, angles, whether a sensor sees a game piece, turns "it didn't work" into "the beam break never tripped." The same two-way channel carries your auto chooser and tuning numbers, so a dashboard is both your window into the robot and your tuning knob. Log liberally.

Going further · optional

A common trap: a value you putNumber() every loop can NOT be tuned from the dashboard, because your code overwrites whatever you typed on the very next loop. For a live-tunable number, publish it once with a default and only read it back, or use WPILib's Preferences, which remembers the value across reboots. Try wiring up a tunable kP that survives a power cycle. Why would a number that resets to its default every time you restart the robot be a headache on competition day?

Show the thinking

On competition day the robot gets power-cycled often, between matches and after fixes, and a value that resets each reboot means every careful tuning change is lost the moment the robot restarts. You would have to retype your kP into the dashboard before every match, which is slow and easy to forget under time pressure, and forgetting it sends the robot out with the wrong gains. Preferences solves this by saving the value to the roboRIO so it survives a power cycle and loads automatically.

Quick check

Pick an answer to see why.

Question 1
What is NetworkTables?
NetworkTables is a live table of named values shared over the network. The robot writes to it, dashboards read from it, and it all stays in sync automatically.
Question 2
Is NetworkTables one-way (robot to dashboard only)?
It's bidirectional. That's what makes live tuning and auto-mode choosers work: the dashboard writes "Shooter/kP" or "Auto/selected" and the robot reads it back, no code redeploy needed.
Question 3
Which tools read NetworkTables to show you the robot's data?
Dashboards like Elastic and Shuffleboard, and the log viewer AdvantageScope, all subscribe to NetworkTables to plot and display what your robot publishes.
