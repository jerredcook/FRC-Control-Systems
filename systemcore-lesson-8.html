Deploying to SystemCore

Image first,
then deploy.

SystemCore runs real-time Linux, so getting code onto it works in two stages: first you "image" the board to set up its operating system, then you send your robot code. If you have ever set up a Raspberry Pi, this will feel familiar.

Alpha preview · this will change

SystemCore is still in alpha and beta testing. The details here are based on what FIRST and WPILib have shared so far and they are not final, so confirm anything important against the official sources on the course page before you rely on it.

Imaging sets up the operating system. Deploying loads your code on top.

Picture a brand-new computer fresh out of the box. Before you can run any of your own programs, the computer needs an operating system installed. Imaging is exactly that step for SystemCore: you flash its Linux operating system onto the board, the same way you write an operating system onto a Raspberry Pi's memory card. Once the board has its operating system, you deploy your robot code on top of it.

The two-stage idea in one picture

Imaging is like putting the operating system on a new laptop before you load your own apps. You do it once to get the machine ready. Deploying is loading your apps, the part you repeat every time you change your code. On a robot you image the board when you set it up, then deploy over and over as you tune and fix things.

How it differs

Not the roboRIO workflow

If you have deployed to a roboRIO, you used a Gradle command, something like ./gradlew deploy, plus a separate imaging tool to set the board up. SystemCore changes this. Because it is a Linux board you image, the setup and deploy steps are new. The exact 2027 commands and tools are still being finalized, so the headline to remember is the shape of the process, not the specific keystrokes.

The exact steps are not final

The 2027 imaging and deploy process is new and still being finalized. Do not memorize a command from a forum post or an old video and assume it is correct. When it is time to set up a real SystemCore, follow the official WPILib and FIRST documentation, because the steps you find there will be the current ones.

What stays the same

Your vendordeps still declare what your robot uses, the same idea as today: the Kraken or TalonFX, your REV gear, your sensors. The board changes and the setup steps change, but the way you tell your project which libraries to pull in is the familiar one.

Going further · optional

On a Raspberry Pi you image the memory card once and then reuse that operating system for many different programs. A robot is similar: imaging is the rare setup step, while deploying is the everyday loop you run dozens of times during a build session. Here is one to think through with a mentor: if imaging is the slow, occasional step and deploying is the fast, frequent one, how would you organize a competition day so a sudden code fix takes seconds, not a full reset of the board?

Show the thinking

The plan is to get all the slow setup out of the way before competition so the only thing left during the day is the fast step. Image the board and confirm it boots and runs known-good code the night before or first thing in the morning, while there is time to fix problems. Then during matches a code change is only a quick deploy, which takes seconds, plus a fast test. Keeping a copy of the last working code on hand also means that if a new change misbehaves, you redeploy the known-good version instead of re-imaging the whole board under pressure.

Question 1
What does "imaging" SystemCore mean?
Imaging flashes the operating system onto the board, much like installing the operating system on a new computer or a Raspberry Pi. You do it to get the board ready, before you deploy your code.
Question 2
How does deploying to SystemCore differ from the roboRIO?
The roboRIO used a separate imaging tool and a Gradle deploy like ./gradlew deploy. SystemCore is a Linux board you image like a Raspberry Pi, so the setup and deploy steps are new and being finalized.
