Docs & Vendordeps

Add libraries,
never get stuck.

RobotPy covers the basics, but your motors come from vendors. A vendor package pulls in their library - and the docs are how you answer your own questions.

Your motor controller speaks its own dialect.

To talk to a Kraken you need CTRE's Phoenix 6 library; for a NEO/SparkMax you need REV's REVLib. With RobotPy these are Python packages. You add one by listing it under requires in your project's pyproject.toml (in the [tool.robotpy] section), then running robotpy sync. That downloads the package and cross-builds it for the roboRIO, and from then on that library's classes are yours to import. Toggle a few below and watch what unlocks.

Only add what you use - extra libraries are extra weight. Most teams run RobotPy plus one motor vendor, maybe PathPlanner and a vision library.

Your lifelines

Reading the docs

You will not memorize every class - nobody does. The skill is finding the answer. Each library has docs with examples and an API reference (the list of every class and method). When you wonder "how do I read the encoder?", search the docs for the class and skim its methods.

How to read an API reference

An API page lists a class's methods: each has a name, what you pass it (parameters), and what it gives back (return type). To find "how do I set a motor's speed?", open the class and scan for a method like set_voltage(volts). Your editor helps too - type a dot after an object and it lists everything you can call.

Going further · optional

Open your pyproject.toml and find where phoenix6 is pinned to a version. That number says which release of the library your project pulls in, and it must line up with the firmware flashed on the device, much like a bike chain that has to match the cassette to mesh. Read about why mismatched library and firmware versions are a classic source of "it worked yesterday" bugs, and how teams keep everyone on the same version.

Show the thinking

The library on your laptop and the firmware on the device have to speak the same protocol, and a version bump can change what calls or fields exist on each side. If one laptop installs a newer phoenix6 but the device firmware is older (or the reverse), calls can fail, behave oddly, or the device may not respond, even though nothing in your own code changed. Teams avoid this by committing pyproject.toml to git so everyone runs robotpy sync to the same pinned version, and by flashing all devices to the firmware that matches it.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
You bought Krakens. What do you need to control them in code?
Krakens use CTRE's TalonFX, so you add the phoenix6 package. (REVLib is for REV's SparkMax/NEO.) RobotPy alone doesn't include vendor motor libraries.
Question 2
How do you add a vendor library?
RobotPy reads the requires list in pyproject.toml; robotpy sync downloads each package and cross-builds it for the roboRIO. No copying source by hand.
Question 3
You don't know how to make a sensor return its value. What's the move?
The docs and API reference list every class and what its methods do. Finding the answer there (or on Chief Delphi) is the core skill - far faster and more reliable than guessing.
