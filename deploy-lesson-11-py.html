The Big Picture

One scheduler,
running everything.

Command-based robots have a heartbeat: a scheduler that, every 20 ms, reads your inputs, runs your active actions, and updates every mechanism. Meet it before the pieces.

Three files, one loop.

A command-based robot is built from a few parts you'll meet over this part. The glue that runs them is the CommandScheduler - called once per loop from robot.py. Here's the cast:

Subsystem

One mechanism - its hardware and state. The nouns of your robot (Elevator, Intake).

Command

An action over time. The verbs - "raise the elevator," "run the intake."

RobotContainer

Where you create the subsystems, build commands, and bind them to buttons. Set up once.

CommandScheduler

The engine. Each loop it runs the active commands and every subsystem's periodic().

The one line that starts it

robot.py hands off to the scheduler

In a command-based project, robot.py is tiny - its main job is to call the scheduler every loop. Everything interesting lives in your subsystems and commands; this just keeps the engine turning:

robot.py Python · RobotPy
def robotPeriodic(self):
    commands2.CommandScheduler.getInstance().run()   # the 3 steps above, every 20 ms

And in RobotContainer - built once at startup - you create the subsystems and wire up the buttons. After that, the scheduler takes over and your robot is alive.

Why command-based at all?

You could stuff everything into teleopPeriodic with a wall of if-statements. Command-based instead splits the robot into named subsystems and reusable commands the scheduler coordinates - so two mechanisms can act at once, actions compose, and your code stays readable as the robot grows. The rest of this part builds those pieces.

Going further · optional

The scheduler runs on the same 20 ms heartbeat as a bike wheel ticking past a sensor: steady, but not free. If one command's execute() does something slow (a long loop, a blocking call), it eats into that 20 ms and the whole robot stutters, a real bug FRC teams hit called loop overrun. Worth pondering: how would you measure where the time goes, and why does that make heavy work like vision belong off the main thread?

Show the thinking

Time the loop itself. RobotPy's watchdog already prints a warning when a loop runs past 20 ms and lists which parts were slow, and you can wrap suspect sections with wpilib.Timer.getFPGATimestamp() before and after to see where the milliseconds go. Vision belongs off the main thread because processing a camera frame can take tens of milliseconds, far more than the 20 ms budget, so running it inline would push every loop into overrun and make the whole robot stutter. Moving it to a separate thread or a coprocessor lets the main loop keep its steady heartbeat and read the latest result when it is ready.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What runs the whole command-based robot each loop?
CommandScheduler.getInstance().run() is the engine - called every 20 ms, it reads inputs, runs active commands, and updates every subsystem. You write the pieces; it runs them.
Question 2
In command-based terms, a Subsystem is a ___ and a Command is a ___.
Subsystems are the nouns - the Elevator, the Intake. Commands are the verbs - raise it, run it. The scheduler coordinates verbs acting on nouns.
Question 3
What is RobotContainer for?
RobotContainer is the setup/wiring file: it instantiates your subsystems and connects controller buttons to commands. It runs once; the scheduler does the per-loop work afterward.
