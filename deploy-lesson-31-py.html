State Machines

One clear
state at a time.

A superstructure has a lot of moving parts. Instead of juggling a dozen booleans, name the states the robot can be in and the events that move between them. Illegal combinations simply can't happen.

The robot is always in exactly one state.

Intake a game piece, hold it, shoot it. Model that as four states, IDLE, INTAKING, LOADED, SHOOTING, with defined transitions between them. Each state says what the mechanisms do; each event either causes a valid transition or is ignored. Press the events and try to break it: you can't shoot from IDLE, because there's no path there.

Superstructure state machineIDLE

A valid event slides the highlight to the next state. An invalid one (shoot while IDLE, note while LOADED) is rejected and logged, the machine refuses to enter an impossible situation. That refusal is the whole point.

In code

An enum and an if/elif

Name the states in an Enum, keep one state attribute, and let each state decide what runs and which events it accepts:

superstructure.py Python · WPILib
from enum import Enum, auto

class State(Enum):
    IDLE = auto()
    INTAKING = auto()
    LOADED = auto()
    SHOOTING = auto()

self.state = State.IDLE

def periodic(self):
    if self.state == State.IDLE:
        self.intake.stop(); self.feeder.stop()
    elif self.state == State.INTAKING:
        self.intake.run()
        if self.beam.has_note(): self.state = State.LOADED
    elif self.state == State.LOADED:
        self.intake.stop()   # hold the piece
    elif self.state == State.SHOOTING:
        self.feeder.run()
        if self.shot_done(): self.state = State.IDLE

def on_intake_button(self):
    if self.state == State.IDLE: self.state = State.INTAKING

def on_shoot_button(self):
    if self.state == State.LOADED: self.state = State.SHOOTING
Why not just booleans?

You could track is_intaking, has_note, is_shooting separately, but then nothing stops the combination "intaking AND shooting", and you write tangled checks to prevent it. A state machine makes the illegal states unrepresentable: there's exactly one state at a time, and only the listed transitions exist. It's the clean way to coordinate a whole robot, and Triggers from Lesson 3.5 can fire the transitions for you.

Going further · optional

The if/elif above mixes two jobs: deciding what each state does, and deciding when to change state. A cleaner pattern gives every state its own on_enter, run, and the transition rules, so adding a new state does not mean editing one giant block. Here is a what-if to chew on: real robots need an escape hatch from any state, for example a driver eject button that forces you back to IDLE from anywhere. Where would that fit, and how do you keep it from quietly hiding a real bug?

Quick check

Pick an answer to see why.

Question 1
What is a state machine?
A state machine lists the states the system can be in and the events that move between them. At any moment it's in one state, which keeps the logic clear and predictable.
Question 2
Why is a state machine better than a pile of separate booleans?
With one state attribute and explicit transitions, contradictory situations simply can't be represented. Separate booleans let you drift into "is_shooting and is_intaking" and force messy guard code.
Question 3
In the machine above, you press shoot while in IDLE. What happens?
Shooting is only reachable from LOADED. With no IDLE-to-SHOOTING transition defined, the event is simply rejected, which is exactly the safety a state machine buys you.
