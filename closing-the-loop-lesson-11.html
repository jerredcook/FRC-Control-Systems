Your First Closed Loop in Code

Make it move,
in code.

Read the sensor, ask a PIDController for a voltage, send it to the motor. The Part I loop - now four real lines of WPILib that actually drive the elevator.

You don't write the PID math.

You learned exactly what P, I, and D do in Part I - and the good news is WPILib already implements them. You hand a PIDController your gains and, every loop, give it two numbers: where you are and where you want to be. It hands back the voltage. Your job is to wire it up correctly and feed it good measurements (the ones you just learned to convert).

Below is a real PID loop running the elevator. Drive it with the gains - it behaves exactly like Lesson 2 and 4, because it is the same controller, now in code.

Cranking kP shrinks the gap but never closes it. Run it, push kP up, and watch the At setpoint readout stay at no while the carriage settles a hair low.

The whole loop

Four lines that matter

Here's the subsystem. The controller is created once with its gains; periodic() runs the loop every 20 ms - read, calculate, apply. That's the entire thing:

Elevator.java - closed-loop position control Java · WPILib
private final PIDController pid = new PIDController(4.8, 0, 0.2);  // kP, kI, kD
private double goalMeters = 0;

public Elevator() {
  pid.setTolerance(0.01);              // "close enough" = 1 cm
}

public void setGoal(double meters) { goalMeters = meters; }

@Override
public void periodic() {
  double measured = getHeightMeters();              // Lesson II.2's conversion
  double volts = pid.calculate(measured, goalMeters);  // P · I · D, done for you
  motor.setVoltage(volts);                       // act
}

Notice what calculate takes: the measurement first, the goal second. It computes the error internally, applies your gains, and returns volts. You never touch the formula - you just feed it the right numbers and use what it returns.

Why it still droops

Pure P needs a leftover error to produce the voltage that holds against gravity - so it settles just short of the target, exactly like Lesson 2. Cranking kP shrinks the droop but invites oscillation. The clean fix isn't more gain; it's feedforward - supplying the gravity voltage directly. That's the very next lesson.

Going further · optional

The D term works by comparing this reading to the last one, like noticing how much closer the wall got between two glances. If your sensor reading is noisy, those tiny jitters between beats look like fast motion to D, and it can fight back hard and make the motor buzz. That is why teams keep kD modest and clean up the measurement first. Try this thought experiment: if every reading were perfect and smooth, would a big kD still cause buzzing?

Show the thinking

With a truly clean reading the noise-driven buzz goes away, because D only sees real motion instead of jitter. But a very large kD can still misbehave: it reacts to the velocity it estimates from one beat to the next, and if it is large enough it can over-correct and oscillate even on a smooth signal. So clean measurement removes the noise problem, and a sane kD removes the rest.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does pid.calculate(measured, goal) return?
It computes the error for you, applies the P/I/D gains, and returns the controller's output - the voltage you hand straight to setVoltage().
Question 2
With only P and D and gravity pulling down, the elevator settles slightly below the target. Why?
It's the Lesson 2 droop: the only way pure P produces a holding voltage is to keep a little error around. Feedforward (next lesson) supplies that voltage directly, so the error can go to zero.
Question 3
How do you let the rest of your code ask "are we there yet?"
setTolerance defines how close counts as "there," and atSetpoint() returns true once the measurement is within that band - handy for ending a command when the move finishes.
