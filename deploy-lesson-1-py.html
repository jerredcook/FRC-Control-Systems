A robot program isn't an app on your laptop.

It's code that runs on a small rugged computer bolted into the robot - the roboRIO. You write it on your laptop, send it across a cable or radio, and from then on it runs on the robot, all by itself, fifty times a second. Your laptop can even walk away.

Three machines do the work, and it helps to know who's who before you write a line:

💻

Your laptop

Where you write the code, in any editor you like. It packages the program and sends it to the robot. After that, it's optional.

🤖

The roboRIO

The robot's brain. It runs your program on a fixed loop, reading sensors and driving motors over the CAN bus.

🎮

Driver Station

The laptop app that enables the robot and shows you what it's doing - your window in while it runs.

The thing that runs

One class is the entry point

When the roboRIO powers on, it looks for one special class - Robot - and runs it. You rarely touch it directly, but it's the seed everything grows from. A brand-new project's robot.py is mostly empty methods waiting for you:

robot.py - the roboRIO's entry point Python · RobotPy
import wpilib


class MyRobot(wpilib.TimedRobot):
    def robotInit(self):        # once, at power-on
        pass

    def robotPeriodic(self):    # every 20 ms, always
        pass

    def teleopPeriodic(self):   # every 20 ms, while you're driving
        pass


if __name__ == "__main__":
    wpilib.run(MyRobot)

TimedRobot is WPILib's base class that calls these methods for you on a clock - robotInit once, the periodic methods every 20 ms. (RobotPy keeps WPILib's original method names, so they stay camelCase even though your own Python code uses snake_case.) You'll spend the course filling them in (and soon, a tidier structure on top called command-based). For now, just know: this is what 'runs on the robot.'

You won't write this from scratch

The RobotPy project template generates robot.py and a starter project for you - you never start from a blank file. The next lesson sets all that up. Today is just the map: laptop writes, roboRIO runs, Driver Station watches.

Going further · optional

The periodic methods run every 20 ms, which gives a budget of 20 ms per loop. What do you think happens if your code inside one loop takes longer than 20 ms to finish, say a slow calculation or a print statement in a tight loop? Look up "loop overrun" and you will find the warning real teams see when their robot starts to stutter.

Show the thinking

If one loop runs long, the next one starts late, so the whole schedule slips behind. WPILib prints a "Loop overrun" warning and your control feels laggy or stuttery because sensor reads and motor updates happen less often than the 20 ms you planned for. The code does not crash, it falls behind, which is why you keep each loop fast: read sensors, decide, set outputs, and push slow or blocking work elsewhere.

Question 1
Where does your robot program actually run?
You write and package on your laptop, then deploy the program to the roboRIO. From then on it runs on the robot itself - the laptop can disconnect entirely.
Question 2
What is the everyday loop of robot programming?
You make a change, build it, deploy it to the robot, run it and watch the result, then go again. Most of the skill is making that loop fast so you can iterate.
Question 3
What's the Driver Station for?
The Driver Station is the app on the laptop that enables/disables the robot, selects the mode (teleop, autonomous, test), and surfaces telemetry and errors - your live window into the running robot.
