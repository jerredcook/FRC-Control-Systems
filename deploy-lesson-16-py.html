Default Commands & Requirements

One owner per
mechanism.

The scheduler's killer feature: two commands can never fight over the same subsystem. And a default command keeps a mechanism doing something useful when nothing else asks.

Every command declares what it needs.

When a command calls addRequirements(drive), it tells the scheduler "I need the drivetrain." The scheduler then guarantees only one command uses Drive at a time - scheduling a new one interrupts the old. Below, try to make two commands fight over the Drive subsystem. You can't.

Schedule Aim then FollowPath - FollowPath interrupts Aim, because both need Drive. RunIntake never disturbs Drive; different requirement. End the Drive command and its default, TeleopDrive, resumes on its own.

In code

setDefaultCommand + requirements

Two lines make this work. addRequirements (you saw it in Lesson 3.3) declares the need; setDefaultCommand gives the subsystem something to run whenever no other command requires it - usually joystick driving:

RobotContainer Python · RobotPy
# runs whenever nothing else requires the drivetrain
drive.setDefaultCommand(
    drive.teleop_drive(driver.getLeftY, driver.getRightX))

# aim needs Drive too, so while it runs it takes over -
# and the default resumes the instant it ends:
driver.x().whileTrue(drive.aim_at_target())
This is why command-based scales

Because the scheduler tracks requirements, you never write "is something else using the drivetrain right now?" logic - it's handled. Add a tenth mechanism and a fiftieth command; nothing can secretly double-drive a motor. That guarantee is the whole payoff of the framework.

Going further · optional

A default command must permanently require its subsystem, so it should never finish on its own (if its isFinished() returns True, it ends and the subsystem sits idle until you reschedule). Here is a sharp gotcha: a default command can be interrupted but resumes the instant the interrupter ends, so what happens if you accidentally give two subsystems the same command instance as their default? Try reasoning it out, then ask why each subsystem needs its own command object.

Show the thinking

One command instance can only be scheduled once at a time, and it would now claim both subsystems as requirements, so the scheduler cannot let it be the default for two mechanisms at once. In practice commands2 rejects this: a command may only be the default for the subsystem it requires, and a single instance requiring two subsystems is not a valid default for either, so you get an error rather than two mechanisms quietly sharing one object. Each subsystem needs its own command object because a command carries its own state and its own requirement set, and two mechanisms running "their" default at the same time are genuinely two separate jobs. Build a fresh command per subsystem, or use a factory that returns a new one each call.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
You schedule a command that requires Drive while another Drive command is running. What happens?
Requirements guarantee a single owner per subsystem. Scheduling a new command that needs Drive interrupts whatever held it (its end(True) runs). No two commands can drive the same motors at once.
Question 2
What is a default command?
setDefaultCommand gives a subsystem a fallback action - usually teleop driving - that runs in the gaps when nothing else needs it, and resumes automatically when a higher-priority command ends.
Question 3
Running a command that requires Intake - does it disturb a command using Drive?
Interruption only happens between commands that share a requirement. Intake and Drive are separate subsystems, so their commands run side by side without conflict.
