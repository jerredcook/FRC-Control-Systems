Hands-on lab

One square,
two ways.

Your XRP drives a square twice: once by counting seconds, once by measuring with its encoders and gyro. Then you sabotage both runs with a coin and watch which one survives.

What you need

Grab these before you start

This lab puts Part 6's autonomous ideas, commands, sequences, sensors, on a real floor.

  • An XRP, assembled, with fresh or freshly charged batteries in its pack.
  • A laptop with WPILib installed, the same setup from Part 1 and the Part 9 capstone.
  • About a meter of clear, flat floor. Hard floor beats carpet.
  • Masking tape to mark the start, and one coin for the sabotage step.
New to the XRP?

The Part 9 capstone explains what the XRP is, what it costs, and how the run-in-simulation-over-WiFi model works. This lab assumes you have one in front of you.

Step 1 · Connect

Get the XRP talking to your laptop

Remember the mental model from the capstone. Nothing is flashed onto the robot. Your program runs in WPILib simulation on your laptop, and a WebSocket link over the XRP's own WiFi network carries motor commands out and sensor readings back, many times per second.

  1. Install WPILib, if you have not already (Part 1).
  2. Make a new project from the XRP template or example in VS Code.
  3. Power on the XRP and connect your laptop to the XRP's WiFi network.
  4. Run the program in simulation: in VS Code, run WPILib: Simulate Robot Code. This connects to the XRP over WiFi.
  5. Open the Simulation GUI, set the robot to Teleop and click Enable, then drive it.
Follow the official guide for the exact clicks

We teach the idea here. Versions and menus change, so follow the official WPILib XRP guide for the exact, current click-by-click setup. It is kept up to date by the people who make the tools.

WPILib XRP guide ↗
What you should see

The Simulation GUI opens on your laptop. Pick your controller under Joysticks, set the robot to Teleop, and click Enable. The template project already drives, so the XRP should respond to the sticks. If it moves, the WiFi link works and you are ready for real code.

Step 2 · The drivetrain

Paste in the subsystem

This is the same XRP Drivetrain the Part 9 capstone uses: XRPMotor on channels 0 and 1, Encoder on DIO 4 and 5 (left) and 6 and 7 (right), and XRPGyro for heading. One addition for the lab: a periodic() readout, so you can watch distance and heading live in the sim GUI while the robot moves.

Drivetrain.java java
import edu.wpi.first.wpilibj.xrp.XRPMotor;
import edu.wpi.first.wpilibj.xrp.XRPGyro;
import edu.wpi.first.wpilibj.Encoder;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;
import edu.wpi.first.wpilibj2.command.SubsystemBase;

public class Drivetrain extends SubsystemBase {
  // XRP drive motors live on channels 0 (left) and 1 (right)
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);

  // The XRP wheel encoders are wired to these DIO channels
  private final Encoder leftEncoder = new Encoder(4, 5);
  private final Encoder rightEncoder = new Encoder(6, 7);

  private final XRPGyro gyro = new XRPGyro();

  // From the XRP example: 60 mm wheels, 585 encoder counts per wheel revolution
  private static final double WHEEL_DIAMETER_M = 0.060;
  private static final double COUNTS_PER_REV = 585.0;

  public Drivetrain() {
    double metersPerCount = (Math.PI * WHEEL_DIAMETER_M) / COUNTS_PER_REV;
    leftEncoder.setDistancePerPulse(metersPerCount);
    rightEncoder.setDistancePerPulse(metersPerCount);
    rightMotor.setInverted(true); // the right side is mirrored
    resetEncoders();
  }

  public void arcadeDrive(double speed, double rotation) {
    leftMotor.set(speed + rotation);
    rightMotor.set(speed - rotation);
  }

  public double getAverageDistanceMeters() {
    return (leftEncoder.getDistance() + rightEncoder.getDistance()) / 2.0;
  }

  public double getHeadingDegrees() { return gyro.getAngle(); }
  public void resetEncoders() { leftEncoder.reset(); rightEncoder.reset(); }
  public void resetGyro() { gyro.reset(); }

  @Override
  public void periodic() {
    // live readout for the lab: watch these numbers in the sim GUI
    SmartDashboard.putNumber("Distance (m)", getAverageDistanceMeters());
    SmartDashboard.putNumber("Heading (deg)", getHeadingDegrees());
  }
}

Drop it into your project's subsystems folder, or replace the template's drivetrain with it, and make sure RobotContainer creates one: private final Drivetrain drivetrain = new Drivetrain();

What you should see

The project builds with no red errors. Now push the XRP forward about a hand-width on the floor and watch Distance (m) climb in the sim GUI's NetworkTables view (about 0.1 for 10 cm). Twist it in place and Heading (deg) changes. If both numbers move, your sensors are wired to your code.

Step 3 · The timed square

Drive a square by counting seconds

First, mark home. Put a piece of tape on the floor and line the XRP up on it, pointing along your first side. Then paste these into RobotContainer. A timed drive could not be simpler. Its whole plan is "push forward for this many seconds, then stop".

Timed building blocks, in RobotContainer java
import edu.wpi.first.wpilibj2.command.Command;
import edu.wpi.first.wpilibj2.command.Commands;

Command timedForward(double seconds, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(speed, 0),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .withTimeout(seconds);
}

Command timedTurn(double seconds, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(0, speed),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .withTimeout(seconds);
}

Command timedSquare() {
  return Commands.sequence(
      timedForward(2.0, 0.5), timedTurn(0.7, 0.5),
      timedForward(2.0, 0.5), timedTurn(0.7, 0.5),
      timedForward(2.0, 0.5), timedTurn(0.7, 0.5),
      timedForward(2.0, 0.5), timedTurn(0.7, 0.5));
}

public Command getAutonomousCommand() {
  return timedSquare();
}

In the sim GUI, set the robot to Autonomous and click Enable. When the square finishes, click Disable. Now tune it by trial: change 2.0 until a side is about half a meter, and change 0.7 until a corner is about a quarter turn. Write your numbers down. You earned them the slow way.

What you should see

The XRP drives four sides and four corners on its own, then stops. It will look mostly like a square. The corners will not be exactly 90 degrees, the sides will not match perfectly, and the finish will miss the tape by a little. Run it twice and the two squares will not even match each other. That wobble is the whole point of this lab.

Step 4 · The measured square

Drive a square by measuring

Now replace guessed seconds with measured sensors. These are the same building blocks the capstone uses: drive until the encoders say the distance is covered, turn until the gyro says 90 degrees. The motors get the same power as before. The only thing that changed is who decides when to stop.

Measured building blocks, in RobotContainer java
Command driveForward(double meters, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(speed, 0),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .beforeStarting(drivetrain::resetEncoders)
    .until(() -> drivetrain.getAverageDistanceMeters() >= meters);
}

Command turnDegrees(double degrees, double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(0, speed),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .beforeStarting(drivetrain::resetGyro)
    .until(() -> Math.abs(drivetrain.getHeadingDegrees()) >= Math.abs(degrees));
}

Command measuredSquare() {
  return Commands.sequence(
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5),
      driveForward(0.5, 0.5), turnDegrees(90, 0.5));
}

public Command getAutonomousCommand() {
  return measuredSquare(); // was timedSquare()
}

This is the difference between pedaling your bike with your eyes shut for exactly ten seconds and watching the trip odometer until it reads 100 meters. The first plan falls apart in a headwind. The second one just takes a little longer and still arrives.

What you should see

The same square, but tighter. Sides come out a consistent half meter, corners land near 90 degrees, and the finish sits much closer to your tape mark. Run it three times and the runs now agree with each other. Measurement is what buys you that repeatability, and you never had to tune a single seconds value.

Step 5 · The coin test

Sabotage the square

Time to prove which square is tougher. Tape a coin flat to the floor about halfway down the first side, right in the path of one wheel. The wheel has to climb over it, which drags that side of the robot for a moment, like riding your bike over a stick with one tire. Run the timed square over the coin, put the robot back on the tape, then run the measured square over the same coin.

What you should see

Timed run: the wheel that hits the coin drags, that side comes up short, and every side after it starts from the wrong place, so the square skews and never recovers. Measured run: the first side takes a beat longer, because the encoder will not say "done" until the wheel has really covered half a meter, and the square still closes. Same disturbance, very different endings.

One honest limit: the encoders fix the distance, not the sideways shove. If the coin knocks the robot's heading mid-side, neither version straightens out until the next gyro turn. Fixing heading while you drive takes a feedback controller, and that is exactly where the Part 9 capstone and the Closing the Loop course pick up.

When it misbehaves

Troubleshooting

Three faults cover most bad lab days.

The problemThe likely causeThe fix
Enable is on but nothing moves Your laptop is still on the regular WiFi instead of the XRP's own network, or the batteries are low Join the network named XRP-something, put in fresh batteries, then restart the simulation
It drives in a curve instead of straight The right motor is not inverted, or the motors are on swapped channels Check rightMotor.setInverted(true) and that left is channel 0, right is channel 1
Distance (m) never changes when it moves The encoder channels are wrong, or the distance-per-pulse setup never ran Encoders are DIO 4 and 5 on the left, 6 and 7 on the right, and setDistancePerPulse belongs in the constructor
Going further · optional

Once the measured square closes, keep going. Swap the square for a triangle, three sides and three 120 degree turns, and see if turnDegrees holds up. Or put the XRP's floor-facing reflectance sensors to work: tape a dark line at the finish and stop on the line instead of at a distance.

Stop on the line, with XRPReflectanceSensor java
import edu.wpi.first.wpilibj.xrp.XRPReflectanceSensor;

private final XRPReflectanceSensor reflectance = new XRPReflectanceSensor();

// readings run from about 0 on a bright floor to about 1 on dark tape
Command driveToLine(double speed) {
  return Commands.runEnd(
      () -> drivetrain.arcadeDrive(speed, 0),
      () -> drivetrain.arcadeDrive(0, 0),
      drivetrain)
    .until(() -> reflectance.getLeftReflectanceValue() > 0.7);
}

Question to chew on: the coin shoved the robot's heading a little sideways in the middle of a side, and even the measured square did not fix that until the next corner. What sensor, and what kind of command, would fix heading during the side?

Show the thinking

The gyro, read continuously while driving. Reset it at the start of the side, then steer with the error: arcadeDrive(speed, -kP * getHeadingDegrees()). If the heading drifts right, the correction steers left, every loop, all the way down the side. That is a proportional feedback controller on heading, the exact idea the Part 9 capstone applies to distance and the Closing the Loop course teaches you to tune properly.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Your timed square looked decent yesterday. Today, with tired batteries, every side comes up short. Why?
A timed drive never measures anything. It runs the motors for 2.0 seconds no matter what, so anything that changes wheel speed, tired batteries, carpet, a coin, changes the distance. The timed square does not even use the encoders, which is exactly its weakness.
Question 2
With the coin taped in one wheel's path, why does the measured square still finish the side at the right distance?
The measured drive's stop condition is getAverageDistanceMeters() >= meters. Drag slows the wheel, so the encoder counts climb slower, so the command simply runs longer until the distance is truly covered. The clock does not decide when to stop. The measurement does.
Question 3
In turnDegrees, which sensor decides when the turn is finished?
turnDegrees resets the gyro, spins in place, and stops when Math.abs(getHeadingDegrees()) reaches the target. The encoders measure how far the wheels roll, not how far the robot has rotated, and the reflectance sensors just stare at the floor's brightness.
