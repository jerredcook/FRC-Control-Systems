Welcome to real code.

Everything in Part I happened in a simulation. Now we move to an actual FRC robot, programmed in Java with WPILib - the standard library every team builds on. Don't worry: you won't write a whole robot today. The goal of this lesson is orientation - to see the shape of a robot program and find the exact spot where the controller you already understand goes.

Remember how every controller in Part I ran "about 50 times a second"? On a real robot that's not a metaphor. The roboRIO - the robot's computer - runs your code on a fixed heartbeat: once every 20 milliseconds, 50 times per second, forever. Measure, decide, act. Measure, decide, act. The whole job of robot code is to fill in what happens on each beat.

The layout

The shape of a project

A WPILib robot is just a folder of Java files. Most teams use the command-based structure, which sorts everything into a few clear roles. Click through the files below to see what each one is for - the whole map fits on one screen.

Two ideas do most of the work: subsystems (the mechanisms) and commands (the actions). Everything else just wires them up.

The important file

A subsystem is a mechanism

A Subsystem represents one piece of hardware - the elevator, an arm, the shooter. It owns that mechanism's motor and sensor, and it has a special method, periodic(), that the robot calls every 20 ms. That method is where the control loop from Part I lives:

Elevator.java - a subsystem Java
public class Elevator extends SubsystemBase {
  private final TalonFX motor = new TalonFX(10);          // the hardware (CAN id 10)
  private final PIDController pid = new PIDController(4.8, 0, 0.2);
  private double goalMeters = 0;

  public void setGoal(double meters) {       // a command calls this
    goalMeters = meters;
  }

  @Override
  public void periodic() {                  // ← the robot runs this every 20 ms
    double measured = motor.getPosition().getValueAsDouble();
    double volts = pid.calculate(measured, goalMeters);   // the Part I loop,
    motor.setControl(new VoltageOut(volts));            // in real code
  }
}

That's the entire idea. periodic() reads where the elevator is, asks the PID controller for a voltage, and applies it - exactly "measure, decide, act," running on the robot's heartbeat. Over the rest of Part II we'll make every line of that method correct and robust.

You don't call periodic() yourself

You never write "run this 50 times a second." You just define periodic(), and WPILib's CommandScheduler - kicked off once in Robot.java - calls it for every subsystem, every beat. Your job is only to say what happens on one beat.

The other half

Commands tell subsystems what to do

If subsystems are the nouns (the elevator), Commands are the verbs (raise the elevator to 1.2 m). A command runs over time, "requires" the subsystems it uses, and is started by something - usually a button press wired up in RobotContainer:

RobotContainer.java - wiring a button to an action Java
private final Elevator elevator = new Elevator();
private final CommandXboxController driver = new CommandXboxController(0);

private void configureBindings() {
  // press A → tell the elevator to go to 1.2 meters
  driver.a().onTrue(elevator.runOnce(() -> elevator.setGoal(1.2)));
}

The command sets the goal; the subsystem's periodic() does the continuous work of getting there. Pressing the button doesn't move the motor directly - it just changes the target the loop is chasing.

Going further · optional

Here is a question to chew on: what happens if your periodic() takes longer than 20 ms to finish, the way a bike chain that skips means you miss a pedal stroke? The roboRIO cannot start the next beat until this one ends, so a slow loop quietly falls behind and your control gets choppy. This is called a loop overrun, and real teams watch for the warning it prints. Try to picture which line in a periodic() would be the slowest, and you have found the first thing to optimize.

Show the thinking

The slow lines are almost always the ones that wait on something outside the code: reading a sensor over a slow bus, asking a vision camera or the network for an answer, or printing to the console with System.out.println. Those block until the answer comes back, so the loop cannot finish until they do. The fixes are to read those values less often, cache them, or hand the heavy work to a separate thread, and to keep logging out of the hot path.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
How often does a standard FRC robot run its periodic code?
The roboRIO runs the loop on a fixed 20 ms beat (50 Hz). Everything - reading sensors, running controllers, setting motors - happens fresh on every one of those beats.
Question 2
In command-based code, where does the PID control loop from Part I actually run?
periodic() is the spot WPILib calls every 20 ms, so the continuous "measure, decide, act" loop lives there. The constructor runs once (setup); Main.java just launches the robot.
Question 3
What's the difference between a Subsystem and a Command?
Subsystems are the nouns (the elevator, its motor and sensor, its loop). Commands are the verbs (go to 1.2 m). Commands set goals; subsystems do the continuous work of reaching them.
