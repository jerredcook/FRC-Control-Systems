Switches & Digital Sensors

Yes or
no.

Some sensors don't give a number, just a true or a false. That one bit is exactly what you need to find a known zero and to tell when a game piece has arrived.

A digital input answers a single yes-or-no question.

A limit switch is pressed or it isn't. A beam break is blocked or it's clear. DigitalInput.get() hands you a boolean, and two jobs lean on it constantly. Homing drives a mechanism slowly until a switch trips, giving it a known zero. Detection watches a beam to know a game piece is in the robot. Run both below.

Homing creeps the elevator down until the switch reads true, then defines that spot as position 0. The beam break flips to BROKEN the instant a game piece crosses it, which is how the robot knows to stop the intake.

In code

One bit, two jobs

A DigitalInput is wired to a DIO port on the roboRIO. You read get(), invert it if the wiring is normally-closed, and build logic on the boolean:

Homing and detection Java · WPILib
DigitalInput bottomLimit = new DigitalInput(0);   // DIO port 0
DigitalInput beamBreak   = new DigitalInput(1);

// invert with ! so "true" always means the active state
boolean atBottom()       { return !bottomLimit.get(); }
boolean hasGamePiece()   { return !beamBreak.get(); }

// homing: creep down until the switch trips, then define zero
public void homingPeriodic() {
  if (!atBottom()) {
    elevator.setVoltage(-1.0);     // slow, gentle
  } else {
    elevator.setVoltage(0);
    elevator.setPosition(0);      // this spot is now "zero"
  }
}
Why home at all? Relative encoders forget

The encoder from Lesson 4.2 only counts change, so when the robot powers on it calls wherever it happens to be "zero." Homing against a physical switch gives an absolute reference every boot, so "go to 20 inches" means the same height every match. Watch the sign on a switch too: many read true when open, so you invert with ! to make true mean "pressed."

Going further · optional

Reading get() once per loop can miss a switch that flips for only a few milliseconds, like a fast game piece zipping past a beam break. WPILib's Counter and asynchronous interrupts catch those quick edges in hardware so nothing slips through between loops. Here is one to chew on: if your robot loop runs 50 times a second, how long could a game piece block the beam and still never be seen by a plain get() check?

Show the thinking

At 50 times a second the loop checks every 20 milliseconds, so anything that blocks the beam, then clears, entirely inside one 20 millisecond gap can be missed. Barely under 20 milliseconds is the worst case: the piece arrives right after one get() and leaves right before the next, so neither check sees it. That is why fast edges are better caught by hardware, with a Counter or an interrupt that latches the event the moment it happens instead of waiting for the next loop.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does a digital input like a limit switch give your code?
Digital inputs are one bit: on or off. A limit switch is pressed or not; a beam break is blocked or clear. That single boolean is all you read with get().
Question 2
Why do teams "home" a mechanism against a limit switch?
A relative encoder only counts change, so it can't know its true position at power-on. Driving to a physical switch gives an absolute reference, so "20 inches" is the same height every time.
Question 3
A switch reads true when it's NOT pressed. How do you make your code's "atBottom" true mean pressed?
Normally-open vs normally-closed wiring flips the sense of get(). A simple ! makes your boolean read the way you think, so true always means "active." Test it on the real robot and invert as needed.
