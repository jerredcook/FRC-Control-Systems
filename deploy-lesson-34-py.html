Unit Testing

Prove the logic
right.

Some code is pure logic: a gear-ratio conversion, a state transition, a deadband. You can test those in milliseconds, with no robot, and lock them down so a future change can't quietly break them.

A test is a tiny claim that runs itself.

"Forty motor rotations through a 10:1 gearbox should equal four mechanism rotations." Write that as an assertion and a test runner checks it instantly, green if it holds, red if it doesn't. Run the suite below: one test is red because of a real bug in the height conversion. Apply the fix and watch it go green.

The red test caught a bug before it ever reached the field: the height conversion forgot to multiply by pi. Apply the fix, re-run, and the whole suite is green. That green is your proof the math is right.

In code

assert, and that's most of it

A pytest test is a function whose name starts with test_ that calls your real code and asserts the result. No robot, no hardware, just logic checked against expectations:

tests/test_conversions.py Python · pytest
import math
import pytest
from conversions import to_mech_rotations, inches


def test_mechanism_rotations():
    # 40 motor turns, 10:1 gearbox -> 4 mechanism turns
    assert to_mech_rotations(40, 10) == pytest.approx(4.0)


def test_height_from_rotations():
    # 2 drum turns, 1.0" drum -> 2 * pi inches
    assert inches(2, 1.0) == pytest.approx(6.283, abs=1e-3)
Tests are a safety net for change

The real value isn't writing the test once, it's the next time someone edits the gear math. If they break it, the test fails on their laptop in seconds, not on the field on Saturday. Test the things that are pure logic, conversions, state machines, command decisions, and run the suite on every commit (CI) so a regression can't sneak past.

Going further · optional

One test that passes proves almost nothing; the trick is choosing inputs that would actually catch a mistake. For the gear math, try adding tests for the tricky edges: zero rotations, a negative direction, a ratio that is not a whole number. A real gotcha lives in that tolerance too: comparing floats needs a small wiggle room (that's what pytest.approx gives you) because 0.1 + 0.2 is not exactly 0.3 in floating point. What happens to your test if you make that tolerance far too loose?

Show the thinking

A tolerance that is too loose makes the test pass even when the answer is wrong, so it stops catching the very bugs it exists to find. If you allow a wiggle of 5 on a gear ratio, a real error of 3 slips through and you ship broken math thinking it was tested. The goal is the smallest tolerance that still survives normal floating point noise, which is why pytest.approx defaults to a tiny relative tolerance: it forgives the rounding in 0.1 + 0.2 but rejects any real mistake. A test you cannot fail on purpose is not really testing anything.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What kind of code is a good fit for unit testing?
Unit tests shine on logic that doesn't need hardware: math, state machines, decision functions. You feed inputs and assert outputs, all in milliseconds on your computer.
Question 2
What's the main long-term benefit of a test suite?
Tests are a safety net. When a future edit breaks the gear math, the suite fails immediately instead of letting the bug reach a match. Run them on every commit.
Question 3
A test is red. What does that tell you?
A red test means the code's output disagreed with the assertion. That's the test doing its job: it points you straight at a bug, which you fix until the test goes green.
