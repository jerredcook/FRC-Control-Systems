Lab · Part 3

Your commands, on
real wheels.

Part 3 gave you subsystems, commands, and default commands. This lab puts all three on the XRP, the roughly $60 real robot from the course capstone: drive it with a gamepad, then break a motor direction on purpose and fix it like a pro.

Before you start

What you need

  • An XRP kit, assembled, with charged batteries in its pack. A tired pack causes weird, random problems, so start fresh.
  • A laptop with WPILib installed, which you set up back in Part 1, and a clear stretch of table or floor.
  • A gamepad, any Xbox-style USB controller. No gamepad? The Simulation GUI can map your keyboard to a virtual joystick, it is just less fun.

One mental model before you touch anything. Your robot program runs on your laptop, in WPILib simulation. The XRP creates its own WiFi network, your laptop joins it, and a WebSocket link carries your motor commands out to the XRP and streams its sensor readings back, many times per second. So "deploy" here means: run in simulation, connected to the real robot over WiFi.

Why a lab

Code that only ever ran in a simulator can hide mistakes that real wheels and a real battery expose in seconds. This lab makes one of those mistakes on purpose, a motor spinning the wrong way, so the first time you meet it, it is in a lab where you already know the fix.

Step 1 · Connect

Power up and connect

  1. In VS Code, run WPILib: Create a new project and pick the XRP example. The example pulls in the XRP vendordep for you, so the XRP classes are ready to import.
  2. Put charged batteries in the XRP and switch it on. It creates a WiFi network named something like XRP-XXXX.
  3. Join that network from your laptop. Your internet drops while you are on it, that is normal.
  4. Run WPILib: Simulate Robot Code. This starts your program on the laptop and connects it to the XRP over WiFi.
What you should see

The Simulation GUI opens on your laptop. Nothing moves yet, and nothing should: the robot is not enabled and you have not written the drivetrain. If the GUI opens and the XRP's power light is on, the link is up and you are ready for Step 2.

Follow the official guide for the exact clicks

Versions and menus change. If any click here does not match your screen, the official WPILib XRP guide has the exact, current setup, kept up to date by the people who make the tools.

Step 2 · Subsystem

Paste the drivetrain subsystem

This is the same SubsystemBase shape from Part 3, filled with the XRP's real hardware: XRPMotor on channels 0 (left) and 1 (right), Encoder on DIO 4 and 5 (left) and 6 and 7 (right), the onboard XRPGyro, and the XRPReflectanceSensor line sensors under the nose. Replace the example's drivetrain with this file.

Drivetrain.java java
import edu.wpi.first.wpilibj.xrp.XRPMotor;
import edu.wpi.first.wpilibj.xrp.XRPGyro;
import edu.wpi.first.wpilibj.xrp.XRPReflectanceSensor;
import edu.wpi.first.wpilibj.Encoder;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;
import edu.wpi.first.wpilibj2.command.SubsystemBase;

public class Drivetrain extends SubsystemBase {
  // XRP drive motors live on channels 0 (left) and 1 (right)
  private final XRPMotor leftMotor = new XRPMotor(0);
  private final XRPMotor rightMotor = new XRPMotor(1);

  // The XRP wheel encoders are wired to these DIO channels
  private final Encoder leftEncoder = new Encoder(4, 5);
  private final Encoder rightEncoder = new Encoder(6, 7);

  private final XRPGyro gyro = new XRPGyro();
  private final XRPReflectanceSensor reflectance = new XRPReflectanceSensor();

  // From the XRP example: 60 mm wheels, 585 encoder counts per wheel revolution
  private static final double WHEEL_DIAMETER_M = 0.060;
  private static final double COUNTS_PER_REV = 585.0;

  public Drivetrain() {
    double metersPerCount = (Math.PI * WHEEL_DIAMETER_M) / COUNTS_PER_REV;
    leftEncoder.setDistancePerPulse(metersPerCount);
    rightEncoder.setDistancePerPulse(metersPerCount);
    rightMotor.setInverted(true); // the right side is mirrored
    leftEncoder.reset();
    rightEncoder.reset();
  }

  public void arcadeDrive(double speed, double rotation) {
    leftMotor.set(speed + rotation);
    rightMotor.set(speed - rotation);
  }

  @Override
  public void periodic() {
    // Runs every 20 ms - a live sensor dashboard for free
    SmartDashboard.putNumber("Heading deg", gyro.getAngle());
    SmartDashboard.putNumber("Left m", leftEncoder.getDistance());
    SmartDashboard.putNumber("Right m", rightEncoder.getDistance());
    SmartDashboard.putNumber("Line left", reflectance.getLeftReflectanceValue());
    SmartDashboard.putNumber("Line right", reflectance.getRightReflectanceValue());
  }
}

Note the one line that already deals with direction: rightMotor.setInverted(true). The right motor is mounted mirror-image to the left, like the pedals on a bike, one crank arm points forward while the other points back. Inverting it once, here, makes "positive" mean "forward" on both sides.

Run WPILib: Simulate Robot Code again. You do not need to enable the robot to test sensors, periodic() runs regardless.

What you should see

Open the NetworkTables view in the Simulation GUI and find your five numbers. Now play: spin a wheel by hand and watch its distance count move. Turn the whole robot in your hand and watch the heading change. Hold it over something dark and the line values climb toward 1, over white paper they fall toward 0. Real sensors, live on your screen.

Step 3 · Default command

Drive it with the gamepad

Lesson 6 of this Part said a subsystem's default command runs whenever nothing else needs it. Teleop driving is the classic default command: whenever the drivetrain is free, feed the sticks to the wheels. Put this in your RobotContainer. The template's Robot.java already builds a RobotContainer, leave that alone.

RobotContainer.java java
import edu.wpi.first.wpilibj2.command.Commands;
import edu.wpi.first.wpilibj2.command.button.CommandXboxController;

public class RobotContainer {
  private final Drivetrain drivetrain = new Drivetrain();
  private final CommandXboxController controller = new CommandXboxController(0);

  public RobotContainer() {
    // Default command: runs whenever nothing else needs the drivetrain
    drivetrain.setDefaultCommand(
        Commands.run(
            () -> drivetrain.arcadeDrive(
                -controller.getLeftY(),   // stick forward reads negative, flip it
                controller.getRightX()),
            drivetrain));
  }
}

That minus sign is not a typo. Gamepads report the Y axis upside down: pushing the stick forward gives a negative number. The minus flips it so forward means forward. Now wire the controller in:

  1. Plug the gamepad into your laptop.
  2. Run WPILib: Simulate Robot Code. In the Simulation GUI, drag your controller from System Joysticks onto Joystick[0].
  3. Set the robot state to Teleop and click Enable.
What you should see

Push the left stick forward and the XRP rolls forward. Right stick left and right spins it in place. Let go of everything and it stops, because the default command is still running and now writes zeros. If it drives backward when you push forward, or the turn feels mirrored, hold that thought, that is exactly what Steps 4 and 5 are about.

Step 4 · Break it

Now break it on purpose

Every team, sooner or later, wires or configures a motor so it runs backward. You are going to cause that bug deliberately, so you recognize it instantly for the rest of your life. Add one wrong line to the Drivetrain constructor:

Drivetrain.java · constructor java
public Drivetrain() {
    double metersPerCount = (Math.PI * WHEEL_DIAMETER_M) / COUNTS_PER_REV;
    leftEncoder.setDistancePerPulse(metersPerCount);
    rightEncoder.setDistancePerPulse(metersPerCount);
    rightMotor.setInverted(true); // the right side is mirrored
    leftMotor.setInverted(true);  // LAB: wrong on purpose. Watch what happens.
    leftEncoder.reset();
    rightEncoder.reset();
}

Restart the simulation, re-drag the joystick, enable Teleop, and push the left stick straight forward.

What you should see

The XRP spins in place instead of rolling forward. Your code still says leftMotor.set(speed + rotation), but the inverted left motor now pushes its wheel backward while the right wheel drives forward. It is like paddling a canoe with one paddle stroking backward: pull hard on both and the boat pirouettes. Notice the lesson hiding here, the code reads correctly while the robot does something absurd. Only real wheels tell you.

Step 5 · Fix it right

Fix it the right way

There are two tempting bad fixes. Swap the motor wires, or sneak a minus sign into arcadeDrive for that one motor. Both make the robot drive today, and both plant a trap: the next person reads the subsystem, believes what it says, and gets burned. Direction is a fact about the hardware, so declare it where the hardware is set up, with setInverted in the constructor:

Drivetrain.java · the honest fix java
rightMotor.setInverted(true);  // the right side is mirrored, flip it once, here
leftMotor.setInverted(false); // the left side is not, and now the code says so

On the XRP the left side is not mirrored, so false is the truthful setting. Writing it explicitly, instead of deleting the line, documents that you thought about it. Restart, enable, and drive.

What you should see

Forward stick, straight roll. A slight drift over a long run is normal for $60 wheels, do not chase it today. The point sticks with you: one call, at the motor, in the constructor. On a full-size FRC robot the exact same rule holds, set inversion once in the subsystem's motor configuration and never scatter minus signs through your commands.

Going further · optional

Three upgrades, in rising order of spice. One: bind a button to a half-speed mode using a Trigger from Lesson 5, hold the bumper and every stick value is scaled by 0.5, handy for tight spaces. Two: use the dashboard heading you already publish to hand-drive the straightest line you can, then check how many degrees you drifted. Three: tape a dark line on the floor and watch the two reflectance values as you push the robot across it by hand, then sketch, on paper only, how a command could steer toward whichever sensor sees the line. That sketch is a line follower, and you already have every ingredient.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
You enable Teleop, push the left stick straight forward, and the XRP spins in place. What is the most likely cause?
Forward in arcade drive means both wheels forward. If one side's inversion is wrong, the forward stick sends one wheel forward and the other backward, like a canoe with one paddle stroking the wrong way, and the robot pirouettes. Spin-instead-of-roll points straight at a direction bug.
Question 2
A drive motor runs backward. What is the right fix?
Swapped wires and scattered minus signs both work today and confuse everyone later. setInverted in the constructor states the truth about the hardware in one place, so every command that drives the subsystem gets the right direction for free.
Question 3
Why is gamepad driving written as the drivetrain's default command?
From Lesson 6: a default command is the subsystem's "when nothing else is happening" behavior. The scheduler runs it whenever no other command requires the drivetrain, which is exactly what teleop driving wants, and it is why the robot stops the moment you let go, the default command keeps running and writes zeros.
