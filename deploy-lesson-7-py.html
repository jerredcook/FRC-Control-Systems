Methods & Logic

Decisions and
reusable code.

A method is a named block of code you can run anytime. Add if/else to make decisions and loops to repeat - and you can express any robot behavior.

A method packages an action behind a name.

You give it inputs (parameters), it does some work, and it hands back a result (the return value). Then you call it by name as many times as you like. Here's one every robot has: a deadband that ignores tiny joystick wiggle so the robot doesn't creep when you're not touching the stick.

deadband(x, tol)which branch runs?

      

        

        

      

      

    

    
The highlighted line is the branch that actually runs. Inside the deadband, the method returns 0; outside, it passes the joystick straight through.

  


  

    
Repeat without copy-paste

    
Loops do a thing N times

    
When you need to do the same thing to four swerve modules, you don't write it four times - you loop. In Python a for loop walks over a sequence directly. range(4) hands you the numbers 0, 1, 2, 3:

    

      
A for loop Python


for i in range(4):              # i takes 0, 1, 2, 3 in turn
    modules[i].configure()      # runs for i = 0, 1, 2, 3  -> four times

    

    
Often you skip the index entirely and loop over the items themselves: for module in modules:. A while loop is the same idea with just a condition: while not at_target: repeats until you've arrived. Both let one block of code handle many cases.

    

      
Methods are how code stays readable

      
Instead of one giant teleopPeriodic with a hundred lines, you write small methods with clear names - deadband(), aim_at_target(), score_high() - and call them. Code you can read is code you can fix at 2 a.m. before a competition.

Going further · optional

      
If you already code, try writing the deadband as a single line with a conditional expression: return 0.0 if abs(x) < tol else x. Then chew on this: a hard cutoff makes the output jump from 0 straight to whatever x is, so the robot lurches the instant you cross the edge. How might you reshape the math so the output eases in smoothly from the deadband instead of snapping? Search "scaled deadband" to compare your idea.

      

        Show the thinking
        
The fix is to rescale the part of the range that is left after the deadband so the output starts at 0 right at the edge and reaches 1 at full stick, instead of jumping. For a positive x past the deadband that is (x - tol) / (1 - tol), mirrored for negative x, which is the "scaled deadband" you will find online. That removes the lurch: crossing the edge now gives a tiny output that grows smoothly. Many teams then square or cube that result for finer control near center, which is a separate shaping choice on top of the same idea.

Question 1

        
What does a method's return value give you?

        

          
          
          
        

        
You call a method with inputs (parameters); it does work and returns a value you can use - like deadband(x, tol) returning the cleaned-up joystick value.

      

      

        
Question 2

        
In the deadband, when does it return 0 instead of x?

        

          
          
          
        

        
The if checks abs(x) < tol. If the joystick is inside that small band, it's treated as zero so noise doesn't drive the robot; outside, x passes through.

      

      

        
Question 3

        
Why use a loop to configure four modules?

        

          
          
          
        

        
A for loop runs the same code for each item. Write it once; it can't drift out of sync the way four copy-pasted blocks can.
