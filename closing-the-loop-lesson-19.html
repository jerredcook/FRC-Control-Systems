The Elevator, End to End

From bare metal
to dialed in.

Every piece you've learned, in order, on one real elevator: configure, characterize, tune, profile, validate. This is the workflow a competitive team runs.

You have all the parts. Now the assembly order.

Tuning a mechanism isn't magic and it isn't guess-and-check - it's a repeatable sequence. Do the steps in the right order and each one makes the next easier. Skip ahead and you'll fight yourself. Here's the whole workflow for the elevator, start to finish.

The workflow

Five steps, in order

1

Configure the device Lesson III.2

Set current limits (protect the motor), brake mode (hold position), and the gear ratio so getPosition() reports mechanism rotations. Do this before anything else.

2

Characterize with SysId Lesson III.5

Run the ramp and step tests, let Tuner X fit the line, and drop kS, kG, kV, kA into Slot0. Now feedforward already does most of the work.

3

Add kP for tracking Lesson I.5

With feedforward in place, raise kP until the elevator hits targets crisply. If it overshoots, add a little kD. You're only trimming the small leftover error.

4

Set Motion Magic limits Lesson III.4

Pick a cruise velocity and acceleration that move fast without slamming. Now every command is a smooth, profiled move from one request.

5

Validate on the robot

Send it to every scoring height, repeatedly. It should nail each one, hold under load, and never oscillate. If it does, you're done - for real.

The capstone

The whole subsystem

Here is everything from Parts II and III, assembled into one real elevator subsystem. The constructor configures the device and loads the SysId gains and Motion Magic limits; goToHeight commands a profiled move. That's a competition-ready mechanism:

Elevator.java - complete Java · Phoenix 6
public class Elevator extends SubsystemBase {
  private final TalonFX motor = new TalonFX(10);
  private final MotionMagicVoltage request = new MotionMagicVoltage(0).withSlot(0);

  public Elevator() {
    var cfg = new TalonFXConfiguration();
    cfg.MotorOutput.NeutralMode = NeutralModeValue.Brake;            // step 1
    cfg.CurrentLimits.StatorCurrentLimit = 60;
    cfg.CurrentLimits.StatorCurrentLimitEnable = true;
    cfg.Feedback.SensorToMechanismRatio = 12.0;
    cfg.Slot0.kS = 0.14;  cfg.Slot0.kG = 0.31;                     // step 2 (SysId)
    cfg.Slot0.kV = 0.115; cfg.Slot0.kA = 0.004;
    cfg.Slot0.kP = 12;    cfg.Slot0.kD = 0.2;                        // step 3 (tracking)
    cfg.Slot0.GravityType = GravityTypeValue.Elevator_Static;
    cfg.MotionMagic.MotionMagicCruiseVelocity = 80;                  // step 4
    cfg.MotionMagic.MotionMagicAcceleration   = 200;
    motor.getConfigurator().apply(cfg);
  }

  public void goToHeight(double rotations) {
    motor.setControl(request.withPosition(rotations));
  }

  public boolean atGoal(double target) {
    return Math.abs(motor.getPosition().getValueAsDouble() - target) < 0.1;
  }
}
The order is the lesson

Feedforward before feedback, always. If you start by cranking kP with no kG, you're using error to fight gravity - exactly the droop from Lesson 2 - and you'll never get it stable. Characterize first; then kP has almost nothing to do, and tuning becomes easy.

Going further · optional

Step 5 says "validate," but a real elevator hides a trap the demo does not: it usually has two motors, a leader and a follower, bolted to the same carriage. If the follower is configured to spin the wrong direction it fights the leader instead of helping, and the giveaway is current, not motion. The carriage might still reach its target while both motors draw far more amps than they should, quietly cooking themselves. Something to ponder: which readout would catch a backwards follower first, position error or stator current, and why?

Show the thinking

Stator current catches it first. The leader can muscle a backward follower into submission, so position error may still look fine and the carriage reaches its target, hiding the fault from anyone watching motion alone. The cost shows up as current: both motors push hard against each other, so stator amps run far above what that move should ever need. Watch each motor's stator current side by side during validation, and a follower spiking opposite the leader, or both pinned near the limit on a light move, is the tell.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What's the right order to tune a mechanism?
Feedforward first. Once kS/kG/kV/kA are doing the heavy lifting, kP only trims a tiny leftover error, so it's easy to tune and stays stable. Doing kP first means fighting gravity with error.
Question 2
After SysId, why does kP usually end up small?
With good feedforward, the predicted voltage is nearly right on its own. kP just cleans up disturbances and model error - so it can be modest, which keeps the loop calm and stable.
Question 3
What does "validate" actually mean for a tuned elevator?
A mechanism is only "done" when it's reliable: it nails each scoring position over and over, holds steady when loaded or bumped, and shows no oscillation. That's what wins matches.
