State-Space & LQR

PID's smarter
sibling.

The math the whole course kept promising. It's less scary than it sounds — and it turns out you've been doing a simple version of it the entire time.

One more way to think about a controller.

So far, control meant "react to the error." State-space zooms out. It describes a system by its state — the few numbers that capture everything about it right now. For our elevator, that's just two: position and velocity. The physics of how the state changes when you apply a voltage is written compactly:

ẋ = A·x + B·u x = state [position, velocity] · u = voltage · A, B = the mechanism's physics

The controller is just as compact: pick a gain for each state and push back. u = K·(reference − x). That's state feedback — and here's the secret it's been hiding.

The reveal

You've been doing this all along

Our state is [position, velocity], so the gain K has two numbers: one multiplying position error, one multiplying velocity. Put names on them and...

u = k₁·(position error) + k₂·(velocity) = kP·error + kD·(velocity) state feedback on [position, velocity] is PD control

k₁ is your kP. k₂ is your kD. PD wasn't a trick — it was state feedback the whole time, on a two-state system. So what does state-space buy you? Two big things: a principled way to choose the gains, and the ability to scale past two states to systems PID can't touch.

One knob, both gains. LQR derives k₁ and k₂ together from a single choice: how much you value tight tracking versus gentle effort.

The optimal part

LQR chooses the gains for you

Trial-and-error tuning is fine for one PID. But what if you have five states? You can't eyeball ten gains. LQR — the Linear-Quadratic Regulator — computes the best K for you from two simple choices:

Q — how much you care about each state's error
R — how much control effort costs you crank Q → tighter, snappier · crank R → gentler, calmer · LQR solves for the K that balances them

That's the whole mental model: tell LQR your priorities, and it hands back the optimal gains. In WPILib it's a few lines built on the mechanism model you'd get from SysId:

State-space control with WPILib Java
// model the elevator as a 2-state system [position, velocity]
var plant = LinearSystemId.createElevatorSystem(
    DCMotor.getKrakenX60(1), massKg, drumRadiusM, gearing);

var lqr = new LinearQuadraticRegulator<>(
    plant,
    VecBuilder.fill(0.02, 0.2),   // Q: tolerable error in position, velocity (smaller = tighter)
    VecBuilder.fill(12.0),       // R: tolerable control effort, in volts
    0.020);

// each loop: u = K (reference − x)
var reference = VecBuilder.fill(goalMeters, 0);     // want this height, zero velocity
var x = VecBuilder.fill(getPosition(), getVelocity());
double volts = lqr.calculate(x, reference).get(0, 0) + gravityFeedforward();
When you'd actually reach for it

For an elevator or flywheel, on-motor PID + feedforward is simpler and plenty. State-space earns its keep on systems with coupled states a single PID can't handle — a double-jointed arm, a flywheel-and-hood shooter, a balancing robot. Same idea, more states. You now know what those teams are talking about.

Quick check

Pick an answer to see why.

Question 1
What is the "state" of a simple elevator?
State is the minimal set of numbers describing the system right now. For an elevator that's position and velocity — together they tell you everything about its motion.
Question 2
How does LQR decide the gains K?
You express your priorities as Q and R; LQR solves for the K that minimizes the resulting cost. Bigger Q → tighter and snappier; bigger R → gentler. No manual gain-guessing.
Question 3
State feedback on [position, velocity] turns out to be…
With a two-state system, K has two entries that multiply position error and velocity — exactly kP and kD. State-space is the general framework; PID is the special case you've been using all along.
