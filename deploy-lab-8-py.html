Hands-on lab

Pro habits on a
desk robot.

Three habits every good robot programmer has: prove the code in simulation first, watch live values instead of guessing, and log every run so the numbers can tell you what happened. This lab practices all three on the XRP.

Before you start

What you need

  • An XRP kit, assembled. The same $60-class desk robot the Part 9 capstone uses.
  • Charged batteries in the XRP. Weak batteries cause the strangest bugs, so start every session with fresh ones.
  • A laptop with RobotPy and WPILib installed (Part 1), and about 45 minutes.

The setup is the same as the capstone: a RobotPy project made from the XRP example, run in simulation on your laptop with robotpy sim, talking to the XRP over its own WiFi network. If any command below has moved in a newer release, the official guide is the source of truth.

Keep the official guide open in a tab

We teach the idea here. Versions and commands change, so follow the official WPILib XRP guide for the exact, current setup, including the precise RobotPy commands. It is kept up to date by the people who make the tools.

Step 1 · Sim first

Run it in simulation, robot still in the box

Pros never test new code on hardware first. A bug on the laptop costs nothing to find. A bug on a moving robot can cost parts. It is the same reason you squeeze your brake levers at the rack before you ride, not halfway down a hill.

  1. Make a new project from the RobotPy XRP example, exactly as in the capstone.
  2. Give it the drivetrain below. It is the capstone's Drivetrain, plus the XRP's two downward-facing line sensors, which we will use in Step 3.
  3. Leave the XRP switched off. Run the program in simulation with robotpy sim.
  4. In the Simulation GUI, drag Keyboard 0 onto Joystick[0], set the robot to Teleop, and click Enable.
drivetrain.py python
from wpilib.xrp import XRPMotor, XRPGyro, XRPReflectanceSensor
from wpilib import Encoder
import math
import commands2


class Drivetrain(commands2.Subsystem):
    WHEEL_DIAMETER_M = 0.060   # 60 mm wheels
    COUNTS_PER_REV = 585.0     # from the XRP example

    def __init__(self):
        super().__init__()
        # XRP drive motors live on channels 0 (left) and 1 (right)
        self.left = XRPMotor(0)
        self.right = XRPMotor(1)
        # Wheel encoders are wired to these DIO channels
        self.left_enc = Encoder(4, 5)
        self.right_enc = Encoder(6, 7)
        self.gyro = XRPGyro()
        # The two downward-facing line sensors on the front edge
        self.reflectance = XRPReflectanceSensor()

        meters_per_count = (math.pi * self.WHEEL_DIAMETER_M) / self.COUNTS_PER_REV
        self.left_enc.setDistancePerPulse(meters_per_count)
        self.right_enc.setDistancePerPulse(meters_per_count)
        self.right.setInverted(True)   # the right side is mirrored
        self.reset_encoders()

    def arcade_drive(self, speed, rotation):
        self.left.set(speed + rotation)
        self.right.set(speed - rotation)

    def average_distance_meters(self):
        return (self.left_enc.getDistance() + self.right_enc.getDistance()) / 2.0

    def heading_degrees(self):
        return self.gyro.getAngle()

    def reset_encoders(self):
        self.left_enc.reset(); self.right_enc.reset()

    def reset_gyro(self):
        self.gyro.reset()

Wire teleop driving as the drivetrain's default command, the same pattern from Part 4:

robotcontainer.py (inside __init__) python
import wpilib
import commands2.cmd as cmd

self.drivetrain = Drivetrain()
self.joystick = wpilib.Joystick(0)
self.drivetrain.setDefaultCommand(
    cmd.run(
        lambda: self.drivetrain.arcade_drive(
            -self.joystick.getRawAxis(1), -self.joystick.getRawAxis(2)
        ),
        self.drivetrain,
    )
)
What you should see

The Simulation GUI opens, the Robot State box says Disabled, and after you click Teleop and Enable your code is running, on the laptop alone. Nothing physical can go wrong here, and that is the point. If robotpy sim errors out or the GUI never appears, fix it now, while the robot is still safely in its box.

Step 2 · Same code, real wheels

Connect the XRP and run it again, unchanged

Now the payoff of the XRP model from the capstone: your program keeps running in simulation on the laptop, and a WebSocket link over the XRP's own WiFi carries motor commands out and sensor readings back, many times per second.

  1. Set the XRP on a small box so its wheels spin in the air for the first test.
  2. Flip the XRP's power switch on and wait about 20 seconds for it to boot.
  3. On the laptop, join the XRP's WiFi network. Its name starts with XRP.
  4. Run robotpy sim again. Same command, zero code changes.
  5. Set Teleop, click Enable, and drive with the same keys as Step 1. Once the wheels behave, put it on the floor and drive for real.
What you should see

The wheels on your desk obey the exact inputs that drove the simulated robot in Step 1. You changed nothing. That is the promise of doing simulation right: the sim and the robot run the same program, so everything you proved in Step 1 carries straight onto hardware.

Step 3 · Watch it live

Publish a value to NetworkTables

NetworkTables is the shared scoreboard every WPILib tool can read. Publish a number once and it shows up live in the Simulation GUI, on dashboards, and in your logs. Think of a bike computer: instead of guessing how fast you climbed the hill, the number is right there while you ride. Add this periodic method to your Drivetrain:

drivetrain.py (add this method) python
from wpilib import SmartDashboard

# add inside the Drivetrain class
def periodic(self):
    # runs every 20 ms, whether or not any command is scheduled
    SmartDashboard.putNumber("Distance (m)", self.average_distance_meters())
    SmartDashboard.putNumber("Heading (deg)", self.heading_degrees())
    SmartDashboard.putNumber("Line left", self.reflectance.getLeftReflectanceValue())
    SmartDashboard.putNumber("Line right", self.reflectance.getRightReflectanceValue())
  1. Run robotpy sim again, connected to the XRP. In the Simulation GUI, open the NetworkTables menu and choose SmartDashboard.
  2. With the robot disabled, roll the XRP forward by hand and watch Distance (m) climb.
  3. Hold the XRP's front edge over something dark, black electrical tape works well, then over white paper. Watch Line left and Line right swing. The reflectance sensors read near 0 over bright surfaces and near 1 over dark ones.
What you should see

Four numbers updating live, about 50 times a second. Roll the robot a hand's width and the distance ticks up by roughly a tenth of a meter. Over tape the line values sit near 1, over paper near 0. You are watching real sensors through the same tables a competition dashboard uses.

Step 4 · Log it, read it back

Record a drive, then study the numbers

Watching live is great while you are standing there. A log is better, because it remembers. DataLogManager records every NetworkTables value, with timestamps, into a .wpilog file, like a bike computer saving your whole ride so you can study the climb afterwards. Two lines in robot.py:

robot.py python
from wpilib import DataLogManager, DriverStation

# add inside your Robot class
def robotInit(self):
    DataLogManager.start()                               # start writing a .wpilog file
    DriverStation.startDataLog(DataLogManager.getLog())  # also record enable state and joystick data
  1. Run robotpy sim connected to the XRP, enable Teleop, and drive one slow lap around your desk. Then disable and close the sim.
  2. When you run in simulation, the log lands in a logs folder inside your project. Find the newest FRC_*.wpilog file.
  3. Open it in AdvantageScope (it installs with WPILib) or the WPILib Data Log Tool, and plot SmartDashboard/Distance (m).
What you should see

A graph of your whole drive: distance flat while you sat disabled, ramping while you drove, flat again when you stopped, with the heading wiggling at every turn. This is how teams debug matches they can barely remember, the log remembers for them. The Data Log Tool can also export CSV, so you can read the same numbers back in a spreadsheet.

When it fights back

Troubleshooting

SymptomLikely causeFix
No XRP network shows up in the WiFi list The XRP is off, still booting, or its batteries are flat Flip the power switch, wait about 20 seconds, and start every session with charged batteries
The sim GUI runs but the robot never moves The laptop is still on your normal WiFi, or the robot is not enabled Join the network whose name starts with XRP, then set Teleop and click Enable
It drove fine yesterday, today it stutters and curves Tired batteries. Weak batteries starve the motors and the WiFi link first Swap in charged batteries before you change a single line of code
Going further · optional

Publish a boolean with SmartDashboard.putBoolean("On line", self.reflectance.getLeftReflectanceValue() > 0.5) and watch it flip as you slide the robot across a taped line. Drop a marker into the log with DataLogManager.log("lap start") at the start of each lap, so you can find your laps in the messages later. Then try this: log the two line values while you slowly slide the XRP sideways across a strip of tape, read the log back, and work out from the numbers alone exactly where the tape was and how wide it is. That is real sensor characterization, the same homework pros do before they write a line follower.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Why did this lab make you run the code in simulation before switching the XRP on?
The sim and the XRP run the exact same program, launched the same way with robotpy sim, so anything you prove in simulation carries straight onto hardware. Finding a bug on the laptop is free. Finding it on a moving robot can cost parts. That is why sim-first is a pro habit, like squeezing your brake levers at the rack instead of halfway down the hill.
Question 2
You call SmartDashboard.putNumber("Distance (m)", ...) inside periodic(). What does that give you?
SmartDashboard writes into NetworkTables, the shared scoreboard every WPILib tool can read. Because periodic() runs every 20 ms, the value updates live, about 50 times a second, and DataLogManager records it into the .wpilog file too. One line of code, visible everywhere.
Question 3
What do DataLogManager.start() and DriverStation.startDataLog(...) get you?
DataLogManager.start() begins writing a .wpilog file and records NetworkTables values automatically. DriverStation.startDataLog(...) adds enable state and joystick data. In simulation the file lands in a logs folder inside your project, and tools like AdvantageScope or the Data Log Tool read it back, like replaying a saved bike ride.
