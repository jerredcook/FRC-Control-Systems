Git & GitHub for Teams

Many hands,
one robot.

A robot is written by a whole team. Git is how a dozen people edit the same code without clobbering each other - and how you undo a mistake instead of panicking.

Git is a time machine and a merge tool.

Every change is saved as a commit - a labeled snapshot you can always return to. You do your work on a branch (a private copy), so you never break the main code. GitHub is the website that holds the team's shared copy. When your branch is ready, you open a pull request so teammates review it before it joins main. Step through the whole flow below.

Predict first
Step through the workflow. After you commit your work on a feature branch, what has to happen before your teammates get those changes on main?
Work on a feature branch stays separate until it is merged into main, and teams usually do that through a pull request so someone reviews it first. Then everyone pulls main to pick it up. Step through the sim and watch the branch rejoin main at the end.
A feature branch, start to finish the team workflow

Small branches, frequent commits, and a pull request for every change - that's how a team ships a robot without chaos on the night before competition.

The commands

The everyday five

You'll type these constantly. Don't memorize - just recognize them; your editor has buttons for all of them too:

Your daily git
git checkout -b intake    # start a branch for your feature
git add .                # stage the files you changed
git commit -m "..."      # save a snapshot with a message
git pull                 # grab teammates' latest first
git push                 # send your branch up to GitHub
Two habits that save seasons

Pull before you push so you build on the team's latest, not an old copy. And never commit secrets or huge files - the generated .gitignore already skips build output, but keep passwords and binaries out of the repo entirely.

Going further · optional

Two people edit the same line of the same file on different branches, then both merge. Git cannot guess who is right, so it stops and hands you a merge conflict, marking both versions with <<<<<<< and >>>>>>> for you to resolve by hand. Read up on how to resolve one, and think about why small, frequent commits and short-lived branches make conflicts rarer and easier to untangle.

Show the thinking

To resolve a conflict, open each marked file, pick the right lines (often a blend of both sides), delete the <<<<<<<, =======, and >>>>>>> markers, then stage and commit. Small, frequent commits and short branches help because two changes only conflict when they touch the same lines, and the longer a branch lives the more lines drift apart from everyone else's work. Merging often means each conflict covers a few lines you still remember, not a giant pile of unfamiliar changes.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What is a commit?
A commit records the state of your files with a message. The full history of commits lets you see what changed, when, and roll back to any point - Git's time-machine superpower.
Question 2
Why work on a branch instead of directly on main?
A branch is a safe sandbox. You commit freely without touching main; when it's solid, a pull request merges it in. main stays deployable the whole time.
Question 3
What's a pull request for?
A pull request (PR) on GitHub says "here's my branch - please review and merge." It's where teammates catch bugs and discuss changes before they reach main. (git pull, a different thing, downloads the latest.)
