Integral Control

Closing
the last gap.

Proportional gets you close. To erase the gap that's left, the controller needs a memory.

Proportional control left something behind.

Last lesson, the elevator climbed toward the line and stopped just short - the steady-state error that pure P can never quite close, because holding against gravity requires some leftover error to generate the voltage. Cranking kP only shrinks the gap; it never erases it. To actually reach the line, the controller needs to notice that a small error has been hanging around - and do something about it.

That's the integral term. It keeps a running total of the error over time - a memory. As long as any error persists, that total keeps growing, and so does the push, until the gap is genuinely gone.

output = kP × error + kI × ∫ error that stretched-S symbol ∫ is just math shorthand for "add it all up over time" - here, the error summed up every loop

kI is the integral gain - how strongly that accumulated history feeds back into the output. Turn it up and the memory pushes harder; turn it to zero and you're back to plain proportional.

With kI at zero, the carriage parks below the line. Add a little and it reaches the line; add a lot and it overshoots before settling.

Why it works

The memory takes over the hold

Watch the numbers as the carriage settles with kI turned up. The I-term climbs and climbs - then levels off at exactly the voltage needed to hold against gravity. The instant the integral is supplying that hold on its own, the proportional term no longer needs any error to do it. So the error falls all the way to zero, and the carriage sits right on the line.

That's the whole trick: the integral remembers the leftover error and accumulates until it can carry the steady load by itself. Steady-state error doesn't get smaller - it gets eliminated.

The cost - overshoot and windup

A memory is slow to change its mind. The integral keeps pushing for a moment even after you reach the line, so it tends to overshoot. Worse: if the target is far away (or the motor is maxed out and physically can't keep up), the integral piles up a backlog of error it couldn't act on - then sails past the target and takes ages to unwind. That runaway accumulation is integral windup, and every real controller needs a way to limit it.

In Python

Adding the integral

By hand, the integral is one new variable that survives between loops, and one new line that grows it. Building on last lesson's controller:

Proportional + integral, by hand Python
kP = 4.0
kI = 1.0
self.integral = 0.0                       # accumulator - persists between loops

def periodic(self):
    error = setpoint - self.encoder.getPosition()
    self.integral += error * 0.02            # 0.02 s = one 50 Hz loop (dt)

    output = kP * error + kI * self.integral
    self.motor.setVoltage(output)

Each loop adds a sliver of the current error - error × dt - onto the running total. While an error lingers, the total grows; once the error hits zero, it stops growing and holds its value. That held value is what keeps the carriage up.

But remember windup. The honest fix is to clamp the accumulator so it can't run away:

Anti-windup - clamp the accumulator Python
self.integral += error * 0.02
self.integral = max(-2.0, min(2.0, self.integral))   # keep the memory bounded

The WPILib version is just last lesson's controller with a nonzero middle gain:

The same thing, with WPILib Python
from wpimath.controller import PIDController

self.controller = PIDController(4.0, 1.0, 0.0)
# kP = 4.0,  kI = 1.0,  kD = 0.0

self.controller.setIZone(0.15)   # only integrate within 0.15 m of target - common anti-windup
Under the hood - taming the integral

WPILib's PIDController accumulates the integral for you. To stop windup, FRC teams reach for one of two knobs: setIntegratorRange(min, max) clamps the term's contribution, and setIZone(zone) only lets the integral build once you're within zone of the target. A small iZone is the usual first move when an integral overshoots on long moves.

Going further · optional

Clamping and iZone are the blunt fixes for windup, but there is a cleverer one used in real controllers called back-calculation: when the motor command hits its voltage limit and saturates, you feed the difference between what you asked for and what the motor could actually deliver back into the integrator to bleed it down, so the memory never charges up on a push the hardware refused to make. Here is the question to chew on: why is windup fundamentally a problem about the gap between the command you send and the command the motor can physically obey, rather than a problem with the integral itself?

Show the thinking

While the motor is pinned at its voltage limit, asking for more does nothing, yet the integrator keeps piling up error as if the extra command were helping. That stored memory is built from a command the hardware never delivered, so it has to be unwound before the output can come back down, which shows up as overshoot. The integral itself is doing exactly its job; the trouble is only the gap between what you asked for and what the motor could physically do, which is why every fix watches that gap rather than the integral alone.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does the integral term accumulate?
Each loop it adds error × dt onto a running total. That growing sum is the controller's memory of how long, and how badly, the target has been missed.
Question 2
How does the integral term erase steady-state error?
As long as any error remains, the integral keeps climbing. It stops only when it's large enough to hold the load on its own - at which point no leftover error is needed, so the error settles at zero.
Question 3
A move to a far target overshoots badly and takes forever to settle. The likely culprit is…
During the long approach the integral accumulates error it can't immediately act on. By arrival it's hugely overcharged, so it overshoots and unwinds slowly. Clamping the integral or using an iZone prevents it.
