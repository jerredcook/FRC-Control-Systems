The last skill isn't code, it's discipline. Commit early, deploy only tested code, and run the same checklist before every match. A good robot with sloppy habits loses to a decent robot with great ones.

A checklist beats a good memory, every time.

Under match pressure, the easiest things get forgotten: the auto never got selected, the battery wasn't charged, someone deployed code that was never tested. A written pre-match checklist makes "ready" a fact, not a feeling. Tick the items and get the robot cleared.

In code

Deploy hygiene

Version control isn't just for big projects. Commit working code before every event so you can always get back to it, and keep the practice field and the match strictly separate:

Build-season workflow git
# before every event: save a snapshot you can return to
git add -A
git commit -m "Tune shooter + 4-piece auto for districts"
git push                          # back it up off the robot
git tag districts-day1            # mark the exact code that ran

# at the field, the rules that win matches:
#  1. deploy ONLY code you tested - the practice field is for testing
#  2. pick your autonomous on the dashboard before EVERY match
#  3. charge and firmly secure a fresh battery
#  4. review the last match's log before changing anything
Never deploy untested code into a real match

The single most expensive habit in FRC is "I'll just push this one fix" minutes before a match, untested. If it has to change, test it on the practice field first, and if there's no time to test, run the last version that worked. A committed, tagged, working build is always your safety net. Slow is smooth, and smooth is fast.

Going further · optional

A tag marks a moment in time, but a branch marks a line of work. Once a robot is at competition, try keeping a stable "competition" branch that only ever gets tested fixes, while bigger experiments live on their own branch you can merge in later. That way a half-finished idea can never accidentally ride along on the build you deploy at the field. Question to chew on: if a fix you pushed between matches makes things worse, what is the fastest way back to the exact code that worked, and how does tagging make that a ten-second job instead of a panic?

Show the thinking

If you tagged the build that worked, for example qual-12-good, getting back is one command: check out that tag and redeploy, and you are running the exact code from that moment. Without a tag you are scrolling through commit history under match pressure trying to guess which one was the good one, which is where mistakes happen. The tag is a named bookmark you made when you knew the code was solid, so recovery is a lookup instead of a search. The habit that makes this work is tagging the build right after each match it performed well, while you still know it was good.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
When should you commit and push your robot code?
Frequent commits and a push before every event mean a crashed laptop or a bad change can't erase your work. A tagged, working build is your safety net.
Question 2
What's the practice field for?
Test changes on the practice field first. Never deploy untested code into a real match; if there's no time to test, run the last version that worked.
Question 3
Why use a written pre-match checklist?
The simplest steps are the easiest to skip when you're rushed. A checklist makes "ready" verifiable, so the robot never takes the field with the auto unselected or a dead battery.
