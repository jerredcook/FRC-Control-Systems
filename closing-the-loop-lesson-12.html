Feedforward in Code

One extra line erases the droop for good. WPILib's ElevatorFeedforward hands the motor the holding voltage up front, so PID can finally relax.

Last lesson left a droop. Here's the fix.

In Part I you learned feedforward: compute the voltage a move needs from physics, before any error appears. WPILib packages that into a class - ElevatorFeedforward - built from the gains you'd measure with SysId (Lesson 7). You create it with kS, kG, kV, kA, ask it for volts, and add that to your PID output. That's the whole pattern.

Flip Feedforward on below and the droop vanishes - not because the gain got bigger, but because the gravity voltage is now supplied directly. Turn it off and Lesson II.3's droop snaps right back.

Toggle it

Feedforward + PID, live

With feedforward on and kG matched to the load, the carriage sits dead on the line - no droop, no integral. Mis-set kG and watch it park high or low; that's feedforward over- or under-guessing gravity.

Predict first
Feedforward is on, but you drag kG well above the gravity it really takes to hold the carriage. Where does the elevator settle?
Feedforward adds kG volts every loop. Set kG too high and it pushes up harder than gravity is pulling down, so the carriage parks above the target until the small PID error nudges back. A flat-on hold only happens when kG matches the real load. Run the sim, push kG up, and watch the carriage settle high.

The code

Two objects, one sum

Create the feedforward and the PID side by side. Each loop, ask both for volts and add them. For a steady hold you pass velocity 0 to the feedforward - and it returns just kG, the gravity-holding voltage:

Elevator.java - feedforward + feedback Java · WPILib
private final ElevatorFeedforward ff = new ElevatorFeedforward(kS, kG, kV, kA);
private final PIDController pid = new PIDController(kP, 0, kD);

@Override
public void periodic() {
  double measured = getHeightMeters();
  double ffVolts = ff.calculate(0);                  // holding → returns kG
  double fbVolts = pid.calculate(measured, goalMeters);  // corrects the rest
  motor.setVoltage(ffVolts + fbVolts);              // predict + correct
}
An arm is almost the same - with one twist

An arm uses ArmFeedforward instead. Gravity's pull on an arm depends on its angle (full at horizontal, zero straight up), so you pass the angle too: armFF.calculate(angleRadians, velocity). It scales kG by cos(angle) for you (cosine is just the math name for that fade from full pull to none; Lesson IV.2 shows it in action). An elevator's gravity is constant, so it needs no angle. Same idea, one extra input.

Going further · optional

The kS term is the voltage needed to break static friction, like the extra shove a bike needs to start rolling from a dead stop. Feedforward adds it in the direction you are moving, which is why kS depends on the sign of velocity, not its size. Here is the tricky part: at a perfect standstill, which way should kS point? Strong feedforward with light PID is what lets a competition arm slam to a new angle and stop dead, instead of crawling in while the error slowly shrinks.

Show the thinking

At a true standstill kS points neither way: its contribution is zero. kS follows the sign of the commanded velocity, and the sign of zero is zero, so the feedforward adds nothing there. That matches the lesson code, where asking the feedforward for a hold at velocity 0 returns kG alone. The job of holding still falls to kG and the PID term, not kS.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
For a static hold, ff.calculate(0) returns approximately…
At zero velocity the kV and kA terms vanish and kS doesn't apply, so the feedforward returns kG - exactly the voltage needed to hang in place. That's what erases the droop.
Question 2
How do you combine feedforward and PID in code?
They're both voltages. Feedforward supplies the predicted bulk; PID adds a correction for whatever's left. You sum them and send the total to the motor.
Question 3
Why does ArmFeedforward need the arm's angle, but ElevatorFeedforward doesn't?
A horizontal arm fights full gravity; a vertical one fights none - so the holding voltage scales with cos(angle). An elevator pulls straight up against the same gravity everywhere, so a single kG covers it.
