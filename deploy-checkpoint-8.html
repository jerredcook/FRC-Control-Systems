Question 1
After a match, a team complains "the shooter never got up to speed." What single logging habit turns that guess into something they can point at on the timeline?
A measured value alone cannot show whether it was right. Logging setpointRpm beside measuredRpm makes the error, the thing a control loop is trying to close, jump out. You replay the .wpilog in AdvantageScope afterward, since a match is over too fast to study live. Review: Lesson 8.1 · Dashboards & Telemetry
Question 2
A team wants to tune their flywheel PID before the robot is even wired. In WPILib simulation, where do the sensor values their PID reads come from?
Simulation runs the exact same subsystem code, but the motor and encoder are replaced by a model. simulationPeriodic() feeds the commanded voltage into FlywheelSim, advances the physics, and reports the result as the measured value, so the loop closes the same way it does on the real robot. Review: Lesson 8.2 · Simulation
Question 3
Which piece of code is the best fit for a fast unit test that needs no robot, and what does a red test tell you?
Unit tests shine on pure logic: conversions, state transitions, decisions. You assert expected against actual with assertEquals. Red means the code's output did not match what you expected, which points you straight at a bug. Run the suite on every commit so regressions cannot sneak back. Review: Lesson 8.3 · Unit Testing
Question 4
Mid-match, several mechanisms push at once and the battery sags below about 6.8 volts, so the roboRIO cuts motor power. This is a brownout. Which line of defense most directly prevents it?
Voltage sag is driven by current. A supply current limit caps how much each motor pulls from the battery, keeping the total draw in a safe range so the bus stays above the brownout threshold. Logging and tests help you understand and prevent problems, but the current limit is what actually keeps the voltage up under load. Review: Lesson 8.4 · Debugging & Robust Code
Question 5
It is five minutes before a match. A teammate wants to push one quick fix that has not been tested. What does deploy discipline say to do?
The most expensive habit in FRC is deploying an untested "one fix" right before a match. The practice field is for testing; the match is not. A committed, tagged, working build is always your safety net, so if there is no time to test, run the last version that worked. Review: Lesson 8.5 · Competition-Ready
