Motion Magic

A smooth move,
from one request.

Part II's motion profile - now built into the motor. Set a cruise speed and acceleration, send a target, and the Kraken plans and tracks the whole trajectory itself.

Last lesson's move was a lunge.

A plain PositionVoltage request snaps the target, so the motor slams toward it and overshoots - just like the unprofiled move back in Lesson II.5. Motion Magic is the onboard version of the trapezoid profile you built there: you give the device a cruise velocity and an acceleration, and it generates a smooth ramp-cruise-slow plan and tracks it with your slot0 gains - all on the motor, from a single request.

Toggle Motion Magic below and feel the difference. Same target, same gains - but one lunges and one glides.

The code

Three numbers and a request

Motion Magic settings live in the config alongside slot0. Set the cruise velocity, acceleration, and (optionally) jerk to round the corners, then send a MotionMagicVoltage request just like a position request:

Configure - Motion Magic limits Python · Phoenix 6
cfg.motion_magic.motion_magic_cruise_velocity = 80    # rotations / sec (top speed)
cfg.motion_magic.motion_magic_acceleration = 160      # rotations / sec^2 (ramp rate)
cfg.motion_magic.motion_magic_jerk = 1600            # rotations / sec^3 (smooths corners)
motor.configurator.apply(cfg)
Commanding a profiled move Python · Phoenix 6
self.request = MotionMagicVoltage(0).with_slot(0)

def set_goal(self, rotations):
    self.motor.set_control(self.request.with_position(rotations))   # device builds + tracks the profile
Motion Magic Expo - even smarter

Regular Motion Magic uses a fixed cruise velocity. Motion Magic Expo instead uses your mechanism's kV and kA to shape an exponential profile that matches what the motor can actually achieve - accelerating hard, then easing as it nears its real top speed. Set motion_magic_expo_k_v and motion_magic_expo_k_a and send a MotionMagicExpoVoltage. No guessing a cruise velocity - the physics picks it.

Going further · optional

Plain Motion Magic asks you to guess a cruise velocity, but a heavy mechanism can never actually reach the speed you picked, so it spends the whole move accelerating and the profile is a fib. Motion Magic Expo skips the guess: from kV and kA it predicts the real speed-vs-time curve, the way a loaded bike trailer eases toward its own top speed instead of a number you wrote down. Try this thought experiment: if you doubled the load on the carriage, which one would need you to re-pick the cruise velocity, and which would adjust on its own?

Show the thinking

Plain Motion Magic is the one you have to re-pick. Doubling the load roughly doubles kA, so the mechanism is now even slower to reach speed, and the cruise velocity you hand-picked is even more of a fib than before, so you would lower it by hand. Expo adjusts on its own once kA reflects the heavier load, because it builds the velocity curve straight from kV and kA instead of from a number you typed. The catch is that Expo only stays honest if those gains are current, so re-characterize after a big mass change.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does Motion Magic add compared to a plain PositionVoltage request?
It's the Part II trapezoid profile, generated and tracked on the device. Instead of snapping to the target, the motor follows a planned ramp-cruise-slow trajectory - gentler and more accurate.
Question 2
What do motion_magic_cruise_velocity and motion_magic_acceleration control?
They define the trapezoid: cruise velocity is the fastest the move will go, acceleration is how hard it ramps up and down. Jerk optionally rounds the corners for an even smoother feel.
Question 3
How is Motion Magic Expo different from regular Motion Magic?
Instead of a fixed cruise velocity you guess, Expo uses kV/kA so the profile naturally respects the motor's real top speed - accelerating hard, then easing in as it approaches it.
