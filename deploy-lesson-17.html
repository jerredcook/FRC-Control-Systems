Part 4 · 01 Talking to Hardware
Motors & Controllers

Command a
motor.

Your code never touches a motor directly. It talks to a motor controller, and how you ask, percent or volts, decides whether the robot behaves the same on a full battery and a tired one.

Between your code and the motor sits a controller.

A motor controller (a TalonFX driving a Kraken, or a SPARK MAX driving a NEO) takes the command from your program and delivers that much electrical power to the motor. The simplest command is open-loop: set(percent) asks for a fraction of whatever the battery has right now, while setVoltage(volts) asks for a specific voltage and holds it as the battery sags. Drag the command up, then drop the battery and watch the difference.

Predict first
Before you run it: you leave the command fixed and lower the battery slider. In which mode does the voltage actually sent to the motor drop?
Percent mode means a fraction of the present battery, so a tired battery gives less output and the motor slows. setVoltage asks for a fixed number of volts and holds it until the battery physically cannot supply them. Run it in percent mode, drop the battery, and watch the output fall, then switch to setVoltage and watch it hold steady.

In percent mode the output is your command times the battery, so it sags when the battery drops. In voltage mode it holds the volts you asked for until the battery physically can't supply them. Set the command to 0 to feel Brake (stops hard) versus Coast (spins down freely).

In code

set, setVoltage, and config

Both vendor libraries follow the same shape: make the controller, command it, configure it. The command you reach for most is setVoltage:

A motor, two vendors Java · Phoenix 6 / REVLib
// CTRE TalonFX driving a Kraken or Falcon
TalonFX motor = new TalonFX(10);
motor.set(0.5);            // open-loop: 50% of whatever the battery has
motor.setVoltage(6.0);     // better: a real 6 volts, sag-compensated

TalonFXConfiguration cfg = new TalonFXConfiguration();
cfg.MotorOutput.NeutralMode = NeutralModeValue.Brake;   // hold still at 0 output
cfg.MotorOutput.Inverted   = InvertedValue.Clockwise_Positive;
motor.getConfigurator().apply(cfg);

// REV SPARK MAX driving a NEO - the other big family
SparkMax neo = new SparkMax(11, MotorType.kBrushless);
neo.setVoltage(6.0);
Why setVoltage wins

A match drains the battery from about 13 volts to 10 or lower. With set(0.5) your mechanism is noticeably weaker at the end of the match than the start, because 50% of 10 volts is less push than 50% of 13. setVoltage(6) delivers 6 volts either way, so the robot feels the same all match. Every closed-loop controller you build later commands volts for exactly this reason.

Going further · optional

There is a ceiling on setVoltage: you can ask for 6 volts all match, but only while the battery can actually supply it. Late in a match a sagging battery may sit near 10 volts under load, so a command for 11 saturates and you silently lose authority right when you need it. Worth pondering: how would you detect that you are commanding more than the battery can give, and why does that make a current limit on each motor part of keeping the whole robot's voltage healthy?

Show the thinking

Compare what you are asking for against what the bus is actually delivering. Read battery voltage from RobotController.getBatteryVoltage() (or the PDH/PDP), and if your commanded voltage is at or above it, you are saturated and cannot get what you asked for. A current limit helps because voltage sag is caused by current draw: every amp pulled through the battery's internal resistance drops the bus voltage for the whole robot. Capping each motor's current keeps any single mechanism, like a stalled arm or a slipping drivetrain, from yanking the bus down and starving everything else, including the roboRIO, which can brown out and reboot. So a per-motor current limit is really a way of protecting the shared voltage that every device depends on.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Why do teams prefer setVoltage(6) over set(0.5)?
set() is a fraction of the live battery voltage, so it weakens as the battery drains. setVoltage() holds the actual volts you asked for, giving repeatable behavior, which is exactly what control loops need.
Question 2
A motor is in Brake neutral mode and you command 0 output. What happens?
Brake mode shorts the motor leads at zero output, resisting motion so the mechanism stops fast and holds. Coast mode lets it spin down freely, which is what you want for, say, a drivetrain that should roll to a stop.
Question 3
What does a motor controller actually do?
The roboRIO runs your code and sends a command over CAN; the motor controller turns that command into the actual voltage and current the motor receives. Your code never wires straight to the motor.
