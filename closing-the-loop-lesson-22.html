Swerve Drive Control

Eight motors,
one intention.

Four wheels that each steer and drive independently. The magic is turning one simple wish - go this way, spin this fast - into eight perfectly coordinated loops.

A swerve drive is just the mechanisms you already know, times four.

Each corner of the robot is a module: a steer motor that points the wheel (position control - like the arm) and a drive motor that spins it (velocity control - like the flywheel). Four modules, eight motors. Nothing new at the motor level - you tuned both kinds in Part IV.

What's new is the layer above: taking what the driver wants - a forward speed, a sideways speed, and a rotation - and computing, for each module, which way to point and how fast to spin. That conversion is called kinematics.

Drive it

One command, four modules

Move the three sliders - forward, strafe, and rotate. Watch each module instantly point where it needs to and spin the right amount. The robot obeys, gliding and turning at once. This is field-oriented control: "forward" means forward on the field, no matter which way the robot faces.

Predict first
You leave forward and strafe at zero and push only the Rotate slider. How do the four module wheels point?
To spin in place, each corner has to drive along the circle around the robot's center, so kinematics points every wheel tangent to that circle, ninety degrees from the line to the center. The four tangents form a pinwheel, and driving them spins the robot without moving it across the field. Run the sim, push only Rotate, and watch the wheels form the circle.

How it works

Three numbers in, eight out

The whole pipeline is three steps, and you've built two of them already:

1 · Chassis speeds

The driver's wish becomes ChassisSpeeds: vx and vy (forward and sideways speed) and omega (the turn rate). Field-oriented control rotates the stick direction by the gyro angle so "forward" stays forward.

2 · Kinematics

SwerveDriveKinematics turns those three numbers into four SwerveModuleStates - an angle and a speed for each corner.

3 · Two loops per module

Each module runs a steer loop (position, like the arm) and a drive loop (velocity, like the flywheel) - the Part IV mechanisms, on the motors.

+ · Odometry

Reading the modules backwards, the robot integrates how far each wheel went to track its pose on the field - where it is and which way it faces.

The control pipeline Java · WPILib + Phoenix 6
// 1. driver wish → field-oriented chassis speeds
ChassisSpeeds speeds = ChassisSpeeds.fromFieldRelativeSpeeds(vx, vy, omega, gyro.getRotation2d());

// 2. kinematics → four module states (angle + speed)
SwerveModuleState[] states = kinematics.toSwerveModuleStates(speeds);

// 3. each module runs its two loops
for (int i = 0; i < 4; i++) modules[i].setDesiredState(states[i]);

// inside a module:
void setDesiredState(SwerveModuleState s) {
  s.optimize(currentAngle);                       // never turn 270° when 90° does it
  steer.setControl(steerReq.withPosition(s.angle.getRotations()));   // position
  drive.setControl(driveReq.withVelocity(toRPS(s.speedMetersPerSecond))); // velocity
}
Two tricks that matter

Optimize: if a module is at 0° and asked for 179°, it's faster to flip to −1° and drive the wheel backward. optimize() does that automatically. Absolute angle: steering needs a FusedCANcoder (Lesson III.2) so each module knows its angle the instant the robot boots - otherwise the wheels start crooked.

Going further · optional

The optimize() step does more than save time, it dodges a genuine trap. Without it, a module asked to swing from 179 degrees to -179 might take the long way around, a near full turn, when a 2 degree nudge plus reversing the drive wheel gets there instantly. Picture steering a bike: you would never spin the bars almost all the way around when a flick the other way points you the same direction. Worth chewing on: across a fast figure-eight, which costs you more lost traction, the steer motors hunting the long way around, or the drive wheels briefly running backward?

Show the thinking

The steer hunting the long way costs you more. While a module slews a big angle, the wheel is pointed wrong for that whole sweep, so it scrubs sideways across the floor and the contact patch fights the direction the robot is actually moving, which is exactly where traction bleeds away. Reversing the drive wheel costs almost nothing by comparison: the wheel still rolls in line with travel, it only spins the other way, and optimize() pairs that small reversal with the short steer move. In a fast figure-eight with constant direction changes, that saved scrub is the difference between holding the floor and sliding.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What two control loops does a single swerve module run?
Each module points its wheel with position control and spins it with velocity control - exactly the two Part IV mechanisms, one per motor. Four modules means eight motors and eight loops.
Question 2
What does kinematics do?
SwerveDriveKinematics turns the three chassis numbers (forward, strafe, rotate) into four SwerveModuleStates. Run it backward - from how far the wheels actually moved - and you get odometry.
Question 3
Why does a swerve module's steering need a FusedCANcoder?
Steering is position control, and the rotor alone forgets its zero at power-off. An absolute CANcoder, fused with the fast rotor, gives the true wheel angle immediately - just like the arm.
