On-Motor Closed Loop

Send a target,
not a voltage.

Put your gains in slot0, fire off a control request, and the Kraken runs the whole PID + feedforward loop on board - at 1 kHz, without your periodic() lifting a finger.

This is the moment it all moves onto the motor.

In Part II you computed volts every loop. Now you do something different: you store your gains in the device's slot0 - all eight of them, kP kI kD and kS kV kA kG - then each time you want to move, you send a control request naming a target. The TalonFX runs feedforward + PID against that target itself, a thousand times a second.

The simulation below holds the elevator with an onboard position loop. Tune the slot0 gains and pick a target - you're not setting voltage anywhere, just telling the motor where to go.

The code

Gains in slot0, a request to move

Configure the eight gains once (these come from SysId - next lesson). Note gravity_type: tell the device whether gravity is constant (elevator) or angle-dependent (arm), and it applies kG correctly:

Constructor - slot0 gains Python · Phoenix 6
cfg.slot0.k_s = 0.12   # volts to overcome static friction
cfg.slot0.k_g = 0.30   # volts to hold against gravity
cfg.slot0.k_v = 0.11   # volts per rotation/sec
cfg.slot0.k_a = 0.01   # volts per rotation/sec²
cfg.slot0.k_p = 4.8    # volts per rotation of error
cfg.slot0.k_i = 0
cfg.slot0.k_d = 0.1
cfg.slot0.gravity_type = GravityTypeValue.ELEVATOR_STATIC
self.motor.configurator.apply(cfg)

Then, to move, build a request once and reuse it. set_control hands the device the target; the onboard loop does the rest:

Commanding a position Python · Phoenix 6
self.request = PositionVoltage(0).with_slot(0)

def set_goal(self, mechanism_rotations):
    self.motor.set_control(self.request.with_position(mechanism_rotations))
Velocity works the same way

A flywheel wants to hold a speed, not a position. Swap in VelocityVoltage and call with_velocity(rps) - same slot0 gains idea, but now kV does the heavy lifting and kP trims. There are torque-current versions too (PositionTorqueCurrentFOC, etc.) for even crisper control. Same pattern every time: configure gains, send a request.

Going further · optional

The device holds more than one gain set: there is a slot0, slot1, and slot2, and a control request can name which to use with with_slot. That lets one mechanism switch gains on the fly, like shifting a bike into a different gear, say gentle gains for holding a game piece and aggressive ones for a fast empty move. Worth chewing on: the PositionVoltage request also carries its own feedforward field you can add per command. When would you want a feedforward that changes move to move instead of one baked into slot0?

Show the thinking

You want a per-command feedforward whenever the extra voltage a move needs depends on something the fixed slot0 gains cannot know ahead of time. The common case is a load that changes: an arm holding a heavy game piece needs more gravity voltage than the same arm empty, so you pass that bonus in the request. The other case is a value computed live each loop, like a feedforward from a motion profile or trajectory that wants a different number at every setpoint. Steady gravity that only varies with angle is already handled by GravityType in slot0, so the per-command field is for what that model misses.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
Where do the gains (kP, kG, kV…) live for an on-motor loop?
You store all eight gains in slot0 and apply() them to the device. From then on, the TalonFX uses them to run feedforward + PID on board against whatever target you send.
Question 2
To make the motor go to a position, your code…
You command a target with a control request like PositionVoltage. The device runs the loop itself - you never compute the voltage. That's the whole point of on-motor control.
Question 3
What does setting gravity_type to ELEVATOR_STATIC vs ARM_COSINE change?
Gravity is constant on an elevator but changes with angle on an arm. gravity_type tells the onboard loop which model to use so kG holds correctly in both cases.
