In code

An auto is a command group

This is Part 3 all over again, now aimed at the field. Sequence the steps, run things in parallel when they overlap, and let the driver pick the routine on the dashboard:

robot_container.py Python · RobotPy + PathPlanner
# an autonomous routine IS a command composition
def score_and_pickup(self):
    return commands2.cmd.sequence(
        self.drive.followPath("ToGoal"),     # drive to the goal
        self.shooter.score_high(),            # score the preload
        self.drive.followPath("ToPickup"),   # go to the next piece
        self.intake.intake_note(),           # pick it up
        self.drive.followPath("ToGoal"),     # back to the goal
        self.shooter.score_high())           # score again

# overlap actions to save time - spin up WHILE driving
self.drive.followPath("ToGoal").alongWith(self.shooter.spin_up())

# let the driver choose the auto from the dashboard
self.chooser.addOption("Four Piece", self.four_piece_auto())
SmartDashboard.putData(self.chooser)

Going further · optional

The very first step of an auto is the easiest to forget: you have to tell odometry where the robot physically starts, or every path will be offset by however far off your assumed start was. PathPlanner can seed the starting pose from the path's first point for you, and a vision fix off an AprilTag can correct it the instant a camera sees a tag. Try this thought experiment: if you place the robot 30 centimeters left of where the code thinks it starts, where does it end up after a flawless three-piece auto, and which single line of code would have fixed it?

Show the thinking

It ends up about 30 centimeters off the whole way through, including at the final scoring spot, because odometry only tracks change from the start, so a wrong starting pose shifts every later pose by that same offset. The auto runs flawlessly relative to where it thinks it began, but in the wrong place. Seeding the correct starting pose fixes it, with a resetPose call (or letting PathPlanner seed from the path's first point), and a vision fix off an AprilTag can correct it mid-auto the moment a camera sees a tag.

Question 1
What is an autonomous routine, in code?
Auto is just the command groups from Part 3 applied to the field: sequence a few path-follow and mechanism commands and the robot runs them on its own.
Question 2
You want the shooter spinning up while the robot is still driving to the goal. How?
Parallel composition runs both at once, so the shooter is at speed the moment the robot arrives. Overlapping actions like this is how teams shave seconds off an auto.
Question 3
How does the drive team pick which auto runs before a match?
A SendableChooser lists your autos on the dashboard (NetworkTables again). The drive team selects one before the match, and the robot reads that choice when autonomous starts.
