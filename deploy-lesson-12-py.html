Subsystems

A home for
one mechanism.

A subsystem wraps one piece of hardware - its motor, its sensor, its state - behind a clean set of methods. It's the noun the rest of your robot talks to.

One subsystem, one mechanism.

An Elevator subsystem owns the elevator's motor, remembers what it's trying to do, and updates itself every loop. Nothing else in your code touches that motor directly - they ask the subsystem. Click each part of a real subsystem to see its job.

Everything the elevator needs is in one file. Commands and the rest of the robot only ever call its public methods - they never reach in and grab the motor.

The big rule

The subsystem owns its hardware

This is the discipline that keeps a robot sane: only the subsystem touches its own motors and sensors. Commands and other code ask through public methods like set_goal(). That way there's exactly one place that controls the elevator, and the scheduler can guarantee two things never fight over it (more in Lesson 3.6).

periodic() is the subsystem's heartbeat

The scheduler calls every subsystem's periodic() once per loop, no matter what command is running. It's the natural home for the control loop - reading the sensor and driving the motor toward the goal. (Exactly the loops from the Control course live here.)

Going further · optional

Try adding a method like at_goal() to your subsystem that returns a boolean (true when the sensor reading is within a small tolerance of the goal). A command can then call it from isFinished(), and a Trigger can watch it. Checkpoint

Quick check

Pick an answer to see why.

Question 1
What does a subsystem contain?
A subsystem models one mechanism: it owns that mechanism's motor and sensor, holds its state (like a goal), and updates itself in periodic(). The Elevator subsystem is everything-elevator.
Question 2
Who is allowed to command the elevator's motor directly?
Encapsulation: the subsystem owns its hardware. Commands ask via methods like set_goal(). One owner means no two pieces of code can secretly fight over the same motor.
Question 3
When does a subsystem's periodic() run?
periodic() runs every 20 ms regardless of what commands are active, so it's the reliable home for a subsystem's continuous control loop.
