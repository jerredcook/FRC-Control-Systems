Docs & Vendordeps

Add libraries,
never get stuck.

WPILib covers the basics, but your motors come from vendors. A vendordep pulls in their library - and the docs are how you answer your own questions.

Your motor controller speaks its own dialect.

To talk to a Kraken you need CTRE's Phoenix 6 library; for a NEO/SparkMax you need REV's REVLib. You add one with a vendordep: in the WPILib VS Code, run Manage Vendor Libraries → Install new (online) and paste the vendor's JSON URL. It drops a file into vendordeps/, and next build, that library's classes are yours to import. Toggle a few below and watch what unlocks.

Only add what you use - extra libraries are extra weight. Most teams run WPILib plus one motor vendor, maybe PathPlanner and a vision library.

Your lifelines

Reading the docs

You will not memorize every class - nobody does. The skill is finding the answer. Each library has docs with examples and an API reference (the list of every class and method). When you wonder "how do I read the encoder?", search the docs for the class and skim its methods.

How to read a Javadoc

An API page lists a class's methods: each has a name, what you pass it (parameters), and what it gives back (return type). To find "how do I set a motor's speed?", open the class and scan for a method like setVoltage(double). Your editor helps too - type a dot after an object and it lists everything you can call.

Going further · optional

Open the Phoenix 6 JSON in vendordeps/ and find the version field. That number pins which release of the library your build pulls in, and it must line up with the firmware flashed on the device, much like a bike chain that has to match the cassette to mesh. Read about why mismatched vendordep and firmware versions are a classic source of "it worked yesterday" bugs, and how teams keep everyone on the same version.

Show the thinking

The library on your laptop and the firmware on the device have to speak the same protocol, and a version bump can change what calls or fields exist on each side. If one laptop updates the vendordep but the device firmware is older (or the reverse), calls can fail, behave oddly, or the device may not respond, even though nothing in your own code changed. Teams avoid this by committing the vendordeps/ JSON to git so everyone pulls the same version, and by flashing all devices to the firmware that matches it.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
You bought Krakens. What do you need to control them in code?
Krakens use CTRE's TalonFX, so you install the Phoenix 6 vendordep. (REVLib is for REV's SparkMax/NEO.) WPILib alone doesn't include vendor motor libraries.
Question 2
How do you add a vendor library?
The WPILib VS Code installs a vendordep from a URL - it adds a small JSON to vendordeps/ and the build pulls in the library. No copying source by hand.
Question 3
You don't know how to make a sensor return its value. What's the move?
The docs and API reference list every class and what its methods do. Finding the answer there (or on Chief Delphi) is the core skill - far faster and more reliable than guessing.
