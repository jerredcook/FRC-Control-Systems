Classes & Objects

Blueprints
and things.

A class is a blueprint. An object is a real thing you stamp out from it. This is how your code models a motor, an arm, a whole robot.

A class describes a kind of thing.

It bundles data (attributes) and actions (methods) together, plus a constructor - the recipe for building one, written as __init__. From a single class you can make as many objects as you want, each with its own data. Here's a tiny Motor blueprint:

motor.py - a class Python
class Motor:
    def __init__(self, name: str, can_id: int):   # ◂ constructor: how to build one
        self.name = name                       # ◂ attribute: this motor's data
        self.can_id = can_id

    def describe(self):                       # ◂ method: an action it can do
        print(f"{self.name} on CAN {self.can_id}")
Try it

One class, many objects

Calling the class like a function - Motor(...) - stamps out an object from it. Make a few Motors below - give each a name and CAN id, and watch each become its own independent thing with its own data.

Each object is a separate box of data made from the same blueprint. left and right are both Motors, but changing one never touches the other.

Why it matters

Objects model hardware

This is the whole reason robot code is built on classes. A TalonFX object is a real motor - you make one with its CAN id, then call methods on it to read and command the physical motor. An Elevator object holds its motor and its state. The robot, in code, is a collection of objects mirroring the metal.

"self" means "this object"

Inside the constructor, self.can_id = can_id means "set this particular object's can_id attribute to the value passed in." It's how each object keeps its own data straight, even though they all ran the same __init__ code. Every method takes self as its first parameter so it knows which object it belongs to.

Going further · optional

Here is a gotcha that bites teams: an object variable does not hold the object itself, it holds a reference to it, like a luggage tag pointing at one bag. So if you write b = a, both names tag the same Motor, and changing b.can_id also changes a.can_id. Try it in the demo logic in your head, then look up "reference vs value" and Python's is operator versus == to see how identity and equality differ.

Show the thinking

Yes, after b = a both names refer to the one Motor, so b.can_id = 5 also shows up as a.can_id, because you copied the tag, not the bag. To get an independent Motor you build a new one with Motor(...) rather than assigning an existing object. The is operator asks whether two names point at the same object (same bag), while == asks whether their contents are considered equal, so here a is b is true; that distinction is exactly the reference-versus-value idea. A name set to None points at nothing, and reaching for an attribute on it raises AttributeError.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What's the difference between a class and an object?
The class describes the kind of thing (its attributes and methods). Each object is one real instance with its own data - like two Motors built from the same Motor class.
Question 2
What does a constructor (__init__) do?
The constructor is the recipe Python runs when you call the class - it runs once per object to initialize that object's attributes (e.g., its name and CAN id).
Question 3
If you make two Motor objects and change one's can_id, what happens to the other?
Objects have independent state. left.can_id and right.can_id are different boxes; changing one leaves the other untouched. That independence is why objects model separate pieces of hardware so well.
