Collections & Enums

Many things,
named states.

Hold a whole group of values at once with arrays and lists - and give a robot a clear set of states with an enum. The last Java you need before building robots.

Sometimes one variable isn't enough.

A swerve drive has four modules; you don't want four separate variables. An array holds a fixed group; a List holds a growable one. You access an array by index - and indexes start at 0:

Arrays and Lists Java
SwerveModule[] modules = { frontLeft, frontRight, backLeft, backRight };
modules[0].setState(...);          // the FIRST module (index 0!)
for (var m : modules) m.stop();    // do something to every module

List<Command> autos = new ArrayList<>();  // growable
autos.add(scoreThenLeave);              // add as many as you like

An enum is a robot's states

An enum is a fixed list of named values - perfect for the states a mechanism moves through. Instead of mystery numbers, you write State.SHOOTING. A switch then does the right thing for whichever state you're in. Step a shooter through its states:

The highlighted chip is the current state; the highlighted case is the code that runs for it. Named states make a robot's behavior readable - and impossible to set to a value that doesn't exist.

  


  

    
Why enums beat numbers

    
Readable and bullet-proof

    
You could track state with an int - 0 for idle, 1 for intaking, 2 for shooting. But then nothing stops you from writing state = 7 (a state that doesn't exist), and if (state == 2) means nothing to a teammate. An enum fixes both: only the listed values are allowed, and the names explain themselves.

    

      
This is a state machine

      
A mechanism that moves through named states based on events - intake until you have a game piece, spin up, then shoot - is a state machine. Enums plus a switch are the simplest way to build one, and you'll use this pattern constantly for shooters, intakes, and superstructures.

Going further · optional

      
A plain enum can carry data too. Try giving your State a constructor field, like a target shooter RPM for each state (SHOOTING(60), IDLE(0)), so state.rpm reads the number instead of a switch. Question to chew on: once states hold their own values, what work is left for the switch to do, and could a Map from state to action replace it entirely?

      

        Show the thinking
        
Once each state carries its own number, the switch is left only with behavior that cannot live on the enum itself, such as calling a motor method or starting a command. If every branch only reads a value, you do not need the switch at all, because state.rpm already gives you the answer. A Map from state to action (an EnumMap) can replace a switch when each state maps to one stored thing, like a Runnable or a setpoint, and it stays in sync because adding a state forces you to add its entry. The switch earns its place when the branches do genuinely different work rather than look up one value.

Question 1

        
What's the index of the first element in a Java array?

        

          
          
          
        

        
Java arrays start at index 0, so modules[0] is the first and modules[3] is the fourth of four. Off-by-one mistakes here are a classic bug.

      

      

        
Question 2

        
Why use an enum for a shooter's states instead of ints?

        

          
          
          
        

        
An enum restricts the value to your listed states (no rogue state = 7) and reads clearly (State.SHOOTING vs 2). That safety and clarity is exactly what a state machine needs.

      

      

        
Question 3

        
You have a growing number of autonomous routines. Which fits?

        

          
          
          
        

        
A List (like ArrayList) holds a variable number of items and grows with add(). Arrays are fixed-size; use a List when the count can change.
