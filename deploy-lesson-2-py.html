Setting Up RobotPy

Your first
project.

Install the tools, generate a project, and meet every file it makes for you - so the folder full of code stops looking like a maze.

You don't install "Python" and start typing.

FRC programming in Python runs on RobotPy - the WPILib library packaged for Python, installed with pip into any editor you already use. One command installs it, and another stamps out a complete starter project. Three steps and you're writing robot code:

1

Install RobotPy

From a terminal, run pip install robotpy. That pulls in WPILib for Python plus the robotpy command line tool. Use any editor you like - VS Code, PyCharm, even a plain text editor.

2

Create a project

Make an empty folder, then run robotpy init inside it. Pick the command-based example as your starting point and fill in your team number when it asks.

3

Open it

That's it - you have a full, runnable robot project. No blank files, nothing missing. Now let's see what's inside.

Two flavors

Command-based vs plain TimedRobot

When you create the project you choose an example to start from. The TimedRobot example is the bare-bones version - just the robot.py methods from the last lesson. The command-based example adds the command-based structure: a RobotContainer, and folders for subsystems and commands.

Pick command-based

Nearly every successful FRC team uses command-based - it keeps a growing robot organized instead of a tangle of if-statements. This whole course teaches it, starting in Part 3. When robotpy init lists examples, the "command-based" one is the clean starting point.

Going further · optional

Try this: open the generated pyproject.toml and find the [tool.robotpy] section. That is where your robot's dependencies and target year live. Running the code in simulation on your laptop with no roboRIO attached is one command away (robotpy sim), which is how you can test logic at home. Skim the comments in that file and see how much of the setup you can follow.

Show the thinking

The [tool.robotpy] section sets your target year and lists the robotpy components your code needs, which is what robotpy sync installs. Running robotpy sim then exercises your logic on the laptop with no roboRIO, so you can test the flow at home. The cost is that sim is not the robot, so a few hardware-only pieces will not behave the same, and you still run robotpy deploy to put real code on the robot. The wider lesson is that pyproject.toml is not magic: it is the one place that declares which libraries and year your project pulls in.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
How do you start a new robot project?
The robotpy init command generates a complete, runnable project from an example - pick command-based and enter your team number. You never start from a blank file.
Question 2
Which files will you actually edit most of the time?
Your robot's behavior lives in subsystems, commands, and the wiring in robotcontainer.py. robot.py and pyproject.toml are scaffolding you rarely touch.
Question 3
Why choose the command-based example over plain TimedRobot?
Command-based gives you subsystems and commands - a clean way to scale from one mechanism to a whole robot without your code turning into spaghetti. It's what this course (and most teams) use.
