Triggers & Button Bindings

Press a button,
fire a command.

Bindings connect the controller to your commands. A button press starts an action - once, or for as long as you hold it. Try the gamepad.

A binding wires a condition to a command.

In RobotContainer you say "when this happens, run that." The most common condition is a button. onTrue fires once when it's pressed; whileTrue runs only while it's held and ends on release. Press the buttons below - watch what fires, and what keeps running.

In code

Bindings live in RobotContainer

You grab the controller, then chain a condition to a command. onTrue, whileTrue, onFalse, and toggleOnTrue cover almost everything:

configureBindings() Java · WPILib
private final CommandXboxController driver = new CommandXboxController(0);

driver.a().onTrue(intake.intakeCommand());      // press → run once
driver.x().whileTrue(shooter.aim());          // held → runs; release → ends
driver.rightBumper().whileTrue(drive.slowMode());
A Trigger doesn't have to be a button

Under the hood, driver.a() is just a Trigger - a boolean condition. Any boolean can be one. Want the controller to rumble when the intake loads a game piece? new Trigger(intake::hasGamePiece).onTrue(rumble()). That's how robots react to sensors, not just buttons.

Going further · optional

Because a Trigger is any boolean, you can combine them with and, or, and negate, the same way you join two conditions in an if-statement. Try binding an action to "X is held and the intake is loaded": driver.x().and(intake::hasGamePiece).onTrue(score()). Question to chew on: a press-and-release happens in an instant, so how does the scheduler notice the exact loop a Trigger flips from false to true, and what would whileTrue do if the condition never goes false?

Show the thinking

Each loop the scheduler reads every bound Trigger and remembers its previous value, so it can compare now against last loop. onTrue fires on the loop where the stored value was false and the new value is true, which is the rising edge, so even a quick press is caught as long as the button is down during one of the 20 ms samples. whileTrue schedules its command when the Trigger becomes true and cancels it when the Trigger becomes false, so if the condition never goes back to false, the command keeps running until something else interrupts it or it finishes on its own. That is why a whileTrue command should be one that ends cleanly or is safe to run continuously.

Checkpoint

Quick check

Pick an answer to see why.

Question 1
What's the difference between onTrue and whileTrue?
onTrue schedules the command once when the condition becomes true. whileTrue runs the command for as long as the condition stays true, interrupting it the moment you let go.
Question 2
Besides buttons, what can a Trigger react to?
A Trigger wraps any boolean condition. new Trigger(intake::hasGamePiece) lets the robot react to a sensor automatically - the same binding machinery, no button required.
Question 3
Where do you set up button bindings?
Bindings are wiring, so they live in RobotContainer's configureBindings(), called once when the robot starts. After that the scheduler watches the triggers for you.
